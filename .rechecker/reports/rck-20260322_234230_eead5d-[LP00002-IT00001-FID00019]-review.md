### BUG: Incomplete Cleanup of `activeOps` on Error in `buildPlugin`
**Severity**: medium
**Location**: `buildPlugin`
**Problem**: The `activeOps` counter is decremented in the `finally` block of the `runBuild` call within `buildPlugin`. However, if an error occurs *before* `runBuild` is called (e.g., during directory creation, manifest writing, or symlink creation), the `activeOps` counter is decremented in the outer `catch` block, but `runBuild` is never called. If `runBuild` is called and then fails, its `finally` block will also decrement `activeOps`. This can lead to `activeOps` being decremented twice for a single operation if an error occurs during the setup phase of `buildPlugin` and then again if `runBuild` is subsequently called and fails. Conversely, if an error occurs in the setup phase and `runBuild` is never called, `activeOps` is correctly decremented once. The issue is the potential for double decrementing if `runBuild` is called and then fails, or if `runBuild` is not called, the `finally` block in `runBuild` will not be executed, leading to `activeOps` not being decremented.

The current structure is:
```typescript
try {
  activeOps++
  // ... setup code ...
  runBuild(...).catch(...).finally(() => { activeOps = Math.max(0, activeOps - 1) })
  return { data: result, status: 202 }
} catch (error) {
  activeOps = Math.max(0, activeOps - 1) // This is the problematic line
  console.error('Error starting plugin build:', error)
  return { error: 'Failed to start plugin build', status: 500 }
}
```
If an error occurs in the setup code *before* `runBuild` is called, `activeOps` is decremented. This is correct.
If `runBuild` is called, its `finally` block will decrement `activeOps`. This is also correct.
The problem is that the outer `catch` block unconditionally decrements `activeOps` if *any* error occurs in the `try` block. If `runBuild` was successfully initiated (meaning the `runBuild(...).catch(...).finally(...)` line was reached), then its own `finally` block will handle the decrement. The outer `catch` should only decrement `activeOps` if `runBuild` was *not* successfully initiated.

**Fix**: The `activeOps` counter should only be decremented in the outer `catch` block if the `runBuild` asynchronous operation was not successfully started. A flag can be used to track this.

```typescript

  try {
    activeOps++
    // ... existing setup code ...

    // Initialize build result

      status: 'building',

    }

      console.error(`Build ${buildId} failed:`, err)
      const r = buildResults.get(buildId)
      if (r && r.status === 'building') {
        buildResults.set(buildId, {
          ...r,
          status: 'failed',
          logs: [err instanceof Error ? err.message : String(err)],
        })
      }
    }).finally(() => {
      activeOps = Math.max(0, activeOps - 1)
    })
    buildStartedSuccessfully = true; // Mark as successfully started

    return { data: result, status: 202 }
  } catch (error) {
    console.error('Error starting plugin build:', error)
    return { error: 'Failed to start plugin build', status: 500 }
  } finally {
    // Only decrement if build was incremented but not successfully started
    if (!buildStartedSuccessfully) {
      activeOps = Math.max(0, activeOps - 1)
    }
  }
```

### BUG: Potential Race Condition in `evictStaleBuildResults`
**Severity**: medium
**Location**: `evictStaleBuildResults`
**Problem**: The `evictStaleBuildResults` function iterates over `buildResults` and performs asynchronous file system operations (`fs.rm`). While `buildResults.delete(id)` removes the entry from the map immediately, the `fs.rm` operation is asynchronous. If the eviction interval is short or many builds are being evicted, multiple calls to `evictStaleBuildResults` could potentially try to delete the same directory, leading to errors or unexpected behavior, although `force: true` mitigates some of this. More critically, if a build is in progress and its `createdAt` timestamp makes it eligible for eviction, it could be removed from `buildResults` while `runBuild` is still trying to update its status. This could lead to `runBuild` failing to find the build ID in `buildResults` and thus not updating the final status, or even worse, if a new build reuses the same ID (though `randomUUID` makes this unlikely), it could corrupt the state of the new build.

**Fix**: Ensure that `runBuild` has completed or failed before attempting to evict a build result. This could involve tracking active build IDs separately or adding a `status` check in `evictStaleBuildResults` to only evict `complete` or `failed` builds. However, a simpler fix for the immediate race with `runBuild` is to ensure `runBuild`'s `finally` block is always executed and that `evictStaleBuildResults` does not remove an actively building item. The current `runBuild` already handles its own `finally` block. The main issue is the `fs.rm` being asynchronous and potentially overlapping. For the `fs.rm` part, it's generally fine with `force: true` as it will just fail silently if the directory is already gone. The more critical part is the `buildResults` map. If a build is `building`, it should not be evicted.

Modify `evictStaleBuildResults` to only evict builds that are `complete` or `failed`.

```typescript
function evictStaleBuildResults(): void {
  const now = Date.now()
  for (const [id, result] of buildResults) {
    // Only evict builds that are complete or failed, not actively building ones
    if (result.status === 'building') {
      continue;
    }
    const age = now - new Date(result.createdAt).getTime()
    if (age > BUILD_TTL_MS) {
      buildResults.delete(id)
      // Best-effort cleanup of build directory
      const buildDir = path.join(BUILDS_DIR, id)
      fs.rm(buildDir, { recursive: true, force: true }).catch(() => {})
    }
  }

  // If still over limit, evict oldest (again, only if not building)
  if (buildResults.size > MAX_BUILD_RESULTS) {
    const entries = [...buildResults.entries()]
      .filter(([, result]) => result.status !== 'building') // Filter out active builds
      .sort((a, b) => new Date(a[1].createdAt).getTime() - new Date(b[1].createdAt).getTime())
    const toRemove = entries.slice(0, entries.length - MAX_BUILD_RESULTS)
    for (const [id] of toRemove) {
      buildResults.delete(id)
      const buildDir = path.join(BUILDS_DIR, id)
      fs.rm(buildDir, { recursive: true, force: true }).catch(() => {})
    }
  }
}
```

### BUG: Unhandled Promise Rejection in `runBuild`
**Severity**: medium
**Location**: `runBuild`
**Problem**: The `runBuild` function is called asynchronously using `.catch()` to handle errors. However, if an error occurs within the `try` block of `runBuild` *before* the `execPromise` call (e.g., `fs.readdir` or `fs.access` throws an error during stats collection), that error will not be caught by the `execPromise`'s `catch` block. Instead, it will propagate as an unhandled promise rejection because the `runBuild` function itself is an `async` function and its returned Promise is not explicitly handled by the caller (`buildPlugin`). While `buildPlugin` has a `.catch()` on the `runBuild` call, it only catches errors from the `runBuild` promise itself, not necessarily errors from within the `runBuild`'s `try` block that might not be part of the `execPromise` chain.

**Fix**: Wrap the entire logic within `runBuild` in a `try...catch` block to ensure all potential errors are caught and the build status is updated correctly.

```typescript
async function runBuild(buildId: string, buildDir: string, manifest: PluginManifest): Promise<void> {
  const existing = buildResults.get(buildId)
  if (!existing) return; // Should ideally not happen if buildId is valid

  try { // Add this try block
    const output = await execPromise(
      path.join(buildDir, 'build-plugin.sh'),
      ['--clean'],
      { cwd: buildDir, timeout: 120000 }
    )

    // Parse output for stats
    const outputPath = path.join(buildDir, manifest.output)
    const stats = { skills: 0, scripts: 0, hooks: 0 }

    try {
      const skillEntries = await fs.readdir(path.join(outputPath, 'skills')).catch(() => [] as string[])
      stats.skills = skillEntries.length

      const scriptEntries = await fs.readdir(path.join(outputPath, 'scripts')).catch(() => [] as string[])
      stats.scripts = scriptEntries.length

      try {
        await fs.access(path.join(outputPath, 'hooks', 'hooks.json'))
        stats.hooks = 1
      } catch {
        stats.hooks = 0
      }
    } catch (statsError) { // Catch errors during stats collection
      console.warn(`Warning: Failed to collect build stats for ${buildId}:`, statsError);
      // Stats collection is non-critical, so we can log and continue
    }

    // Atomic replacement: avoids torn reads from polling clients
    buildResults.set(buildId, {
      ...existing,
      status: 'complete',
      outputPath,
      logs: output.split('\n'),
      stats,
    })
  } catch (error: unknown) { // Catch any error within runBuild
    const message = error instanceof Error ? error.message : String(error)
    const stderr = (error as any)?.stderr
    const logs = [message]
    if (stderr) logs.push(...String(stderr).split('\n'))

    // Atomic replacement
    buildResults.set(buildId, {
      ...existing,
      status: 'failed',
      logs,
    })
  }
}
```

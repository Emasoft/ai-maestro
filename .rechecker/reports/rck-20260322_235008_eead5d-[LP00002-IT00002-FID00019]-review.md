### BUG: Incorrect handling of `result.error` in `sendServiceResult`
**Severity**: high

**Problem**: The `sendServiceResult` function checks `result.error && !result.data` to determine if an error response should be sent. However, if `result.data` is `null` or `undefined` (which can be valid for some successful operations, e.g., a DELETE operation that returns no content but is successful), and `result.error` is also `null` or `undefined` (meaning no error occurred), the condition `result.error && !result.data` would evaluate to `false`. More critically, if `result.error` is a falsy value (like an empty string or `0`), but `result.data` is also falsy, it might incorrectly fall into the success path. The primary issue is that `result.error` being truthy should *always* trigger the error path, regardless of `result.data`.
**Fix**: The condition should simply be `result.error`. If `result.error` exists, it's an error. If `result.error` does not exist, it's a success, and `result.data` should be sent (even if `null`/`undefined`, which `sendJson` handles by sending `{}`).

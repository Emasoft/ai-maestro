# Loop 2 Correctness Review — BuildAction.tsx

### BUG: AbortController ref not guarded before nulling in catch block
**Severity**: high
**Location**: polling `useEffect` catch block — code that does `pollAbortRef.current = null` after aborting a tick's AbortController
**Problem**: When the poll abort is cancelled (tickAbortCtrl.abort() called), the catch block runs `pollAbortRef.current = null`. But by the time the catch executes, `pollAbortRef.current` may already point to a NEWER tickAbortCtrl (from the next tick). Nulling it unconditionally breaks the next tick's abort chain.
**Fix**: Guard the null assignment: only null `pollAbortRef.current` if it still points to the exact same controller that was aborted: `if (pollAbortRef.current === tickAbortCtrl) { pollAbortRef.current = null; }`

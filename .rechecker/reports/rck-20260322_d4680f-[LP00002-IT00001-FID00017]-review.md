# Loop 2 Correctness Review — remote-install.sh

### BUG: CRITICAL — Path traversal in cleanup() via glob pattern matching
**Severity**: critical
**Location**: `cleanup()` function — condition `[[ "$INSTALL_DIR" == "$HOME"/* ]]`
**Problem**: Glob pattern matching on raw path strings allows path traversal. An attacker could set INSTALL_DIR to `$HOME/../../../etc/ai-maestro` which still matches `$HOME/*` as a glob pattern while resolving outside $HOME. On partial install failure in non-interactive mode, this triggers `rm -rf` on arbitrary paths.
**Fix**: Replace glob check with path canonicalization:
```bash
resolved=$(cd -P "$INSTALL_DIR" 2>/dev/null && pwd 2>/dev/null)
[[ -n "$resolved" && "$resolved" == "$HOME"/* ]]
```

### BUG: Misleading Yarn success message printed even when install failed
**Severity**: medium
**Location**: `act2_install_prerequisites()` — `maestro_ok "Yarn installed"` after `npm install -g yarn || { maestro_warn "Could not install Yarn"; }`
**Problem**: `maestro_ok "Yarn installed"` is printed unconditionally even if the install failed (the `|| warn` suppresses set -e). The script proceeds to use `yarn` in subsequent steps which fails.
**Fix**: After the install attempt, verify yarn is available:
```bash
if command -v yarn &>/dev/null; then
    maestro_ok "Yarn installed"
else
    maestro_fail "Yarn install failed — cannot continue"
    exit 1
fi
```

### BUG: tmux new-window/new-session prompt argument not properly quoted
**Severity**: high
**Location**: `act5_grand_finale()` — tmux commands passing `$AI_TOOL` and `$INITIAL_PROMPT`
**Problem**: When launching the AI tool in a tmux window/session, `$INITIAL_PROMPT` is not properly escaped for tmux. Embedded quotes in the prompt break the command.
**Fix**: Use the pattern: `"$AI_TOOL" "${INITIAL_PROMPT//\"/\\\"}"` — replacing double quotes inside the prompt with escaped quotes.

### BUG: lsof kill only handles single PID
**Severity**: medium
**Location**: `act4_start_and_register()` — service restart without pm2
**Problem**: `lsof -ti:"$PORT" 2>/dev/null | kill` only handles a single PID. Multiple processes on the same port fail.
**Fix**: Pipe to xargs: `lsof -ti:"$PORT" 2>/dev/null | xargs -r kill || true`

### BUG: PORT variable not validated as numeric
**Severity**: medium
**Location**: `main()` or argument parsing — PORT variable assignment
**Problem**: PORT is used in numeric contexts (`lsof -ti:"$PORT"`, firewall rules) but is never validated to be a number in range 1024-65535.
**Fix**: Add validation after PORT is set:
```bash
if ! [[ "$PORT" =~ ^[0-9]+$ ]] || [ "$PORT" -lt 1024 ] || [ "$PORT" -gt 65535 ]; then
    maestro_fail "PORT must be a number between 1024 and 65535"
    exit 1
fi
```

### BUG: Multiline gw_list breaks sed substitution
**Severity**: high
**Location**: `act3_clone_and_build()` or config generation — code that builds `gw_list` and substitutes `{{ACTIVE_GATEWAYS_LIST}}`
**Problem**: `gw_list` is built with embedded newlines (using `\n` or actual newlines). Passing a multiline variable to `sed -i "s|{{ACTIVE_GATEWAYS_LIST}}|${gw_list}|g"` breaks sed because sed cannot handle newlines in the replacement string.
**Fix**: Use `awk` instead of `sed` for the substitution:
```bash
awk -v gateways="$gw_list" '{gsub(/\{\{ACTIVE_GATEWAYS_LIST\}\}/, gateways)}1' "$config_file" > "${config_file}.tmp" && mv "${config_file}.tmp" "$config_file"
```

### BUG: Tool availability check uses wrong logic for --skip flags
**Severity**: high
**Location**: `act3_clone_and_build()` — tool_flags construction loop
**Problem**: The tool-skip flags logic adds `--skip-X` when a tool IS found (inverted condition). Should add `--skip-X` when the tool is NOT found.
**Fix**: Change condition from `[ -f "$tool_path" ]` to `[ ! -f "$tool_path" ]` (or `! command -v "$tool" &>/dev/null`) when deciding to add skip flags.

### BUG: _install_npm_global lacks sudo fallback for system Node
**Severity**: medium
**Location**: `_install_npm_global()` helper function
**Problem**: `_install_npm_global` runs `npm install -g "$pkg"` without checking if the global npm directory is writable. On system Node installations (/usr/local), this fails with EACCES.
**Fix**: Mirror the Yarn install logic: check if `npm config get prefix`/lib is writable; if not, use `sudo npm install -g "$pkg"` with a non-interactive check similar to the Yarn block.

# Loop 2 Correctness Review — plugin-builder-service.ts

### BUG: Dead error handler on runBuild() swallows build errors silently
**Severity**: high
**Location**: `buildPlugin()` function — the `.catch(err => { ... })` block chained to `runBuild(...)`
**Problem**: A `.catch()` block is attached to `runBuild()` but the outer code also uses `await`. This means any rejection from `runBuild()` is caught by the `.catch()` and handled (or swallowed) before the `await` sees it, creating dead code and hiding errors from the caller.
**Fix**: Remove the `.catch(err => {...})` block from the `runBuild(...)` call. Let the `await` propagate errors normally to the outer try-catch.

### BUG: validateBuildConfig does not validate core skill paths for path traversal
**Severity**: critical
**Location**: `validateBuildConfig()` function — core skills validation block
**Problem**: When validating `core` type skills, the `skillPath` property (which maps to a filesystem path) is not checked for path traversal sequences (`..`, absolute paths, or null bytes). A malicious input with `skillPath: "../../etc/passwd"` would pass validation.
**Fix**: Add path traversal validation for core skill paths:
```typescript
if (skill.type === 'core' && skill.skillPath) {
    if (!SAFE_PATH_SEGMENT_RE.test(skill.skillPath) || skill.skillPath.includes('..') || path.isAbsolute(skill.skillPath)) {
        return { valid: false, error: `Invalid core skill path: ${skill.skillPath}` }
    }
}
```

### BUG: validateBuildConfig does not validate marketplace skill fields for path traversal
**Severity**: critical
**Location**: `validateBuildConfig()` function — marketplace skills validation block
**Problem**: For `marketplace` type skills, `skill.marketplace`, `skill.plugin`, and the skill name component of `skill.id` (format: `marketplace:plugin:skillName`) are used to construct filesystem paths in `generateManifest()`. None of these are validated against path traversal.
**Fix**: Add validation for marketplace skill fields:
```typescript
if (skill.type === 'marketplace') {
    const SAFE_RE = /^[a-zA-Z0-9_.-]+$/
    if (skill.marketplace && !SAFE_RE.test(skill.marketplace)) {
        return { valid: false, error: `Invalid marketplace name: ${skill.marketplace}` }
    }
    if (skill.plugin && !SAFE_RE.test(skill.plugin)) {
        return { valid: false, error: `Invalid plugin name: ${skill.plugin}` }
    }
    // Validate skill name component from id
    const skillName = skill.id?.split(':')[2]
    if (skillName && !SAFE_RE.test(skillName)) {
        return { valid: false, error: `Invalid skill name in id: ${skillName}` }
    }
}
```

### BUG: validateGitUrl allows URLs with embedded credentials
**Severity**: critical
**Location**: `validateGitUrl()` function
**Problem**: The function validates that a URL is a GitHub URL and uses HTTPS, but does not reject URLs with embedded credentials like `https://user:password@github.com/org/repo`. These credentials would be stored in config files and logged.
**Fix**: After parsing the URL, check for embedded credentials:
```typescript
const parsed = new URL(gitUrl)
if (parsed.username || parsed.password) {
    return { valid: false, error: 'Git URL must not contain embedded credentials' }
}
```

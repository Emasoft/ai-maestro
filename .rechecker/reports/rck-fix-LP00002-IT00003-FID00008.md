# Fix Report: FID00008

**File:** `components/plugin-builder/BuildAction.tsx`

## BUG 1 (critical) — Fixed

**Location:** `handlePush` function, line 117

**Before:**
```ts
if (!forkUrl.trim() || !result.manifest) return
```

**After:**
```ts
if (!forkUrl.trim() || !result?.manifest) return
```

**Root cause:** `result` state is typed as `PluginBuildResult | null` and initialized to `null`. Accessing `.manifest` directly on a `null` value throws a TypeError at runtime. The optional chaining operator (`?.`) short-circuits safely when `result` is `null`, making the guard correct and defensive.

## BUG 2 (medium) — Fixed

**Location:** cleanup-only `useEffect`, lines 41-45

**Before:**
```ts
}, [clearPoll])
```

**After:**
```ts
}, [])
```

**Root cause:** The sole purpose of this `useEffect` is to register a cleanup function that runs on component unmount. `clearPoll` is defined with `useCallback([], [])` — it has no external dependencies and its reference never changes across renders. Including it in the dependency array caused the effect to re-run unnecessarily on any render where React detected a new reference (though stable in practice, it is semantically incorrect). An empty dependency array `[]` correctly expresses mount/unmount-only intent and eliminates any future risk if `clearPoll`'s memoization is ever changed.

## Verification

Both changes verified by re-reading the affected lines after edit. No other instances of `result.manifest` (without optional chaining) or similar cleanup `useEffect` patterns with stale dependencies were found elsewhere in the file.

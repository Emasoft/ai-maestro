# Fix Report: FID00011 — SkillPicker/RepoScanner repo-key ref mismatch

## Summary

All 4 fixes applied successfully to resolve the `ref` omission from repo skill keys.

## Fixes Applied

### Fix 1 — SkillPicker.tsx: `getSkillKey` repo case now includes `ref` (HIGH)
- **File**: `components/plugin-builder/SkillPicker.tsx` line 277
- **Before**: `return \`repo:${skill.url}:${skill.skillPath}\``
- **After**: `return \`repo:${skill.url}:${skill.ref}:${skill.skillPath}\``
- **Root cause**: `ref` was part of `PluginSkillSelection` but ignored in the key, so two skills from the same URL/path on different branches would collide.

### Fix 2 — RepoScanner.tsx: `key` for `selectedSkillKeys.has()` includes `ref` (HIGH)
- **File**: `components/plugin-builder/RepoScanner.tsx` line 130
- **Before**: `const key = \`repo:${url}:${skill.path}\``
- **After**: `const key = \`repo:${url.trim()}:${ref.trim() || 'main'}:${skill.path}\``
- **Root cause**: Key did not include `ref` and did not trim `url`, causing `has()` lookups to never match keys produced by `getSkillKey()`.

### Fix 3 — RepoScanner.tsx: `onRemoveSkill` called with fully-qualified key (HIGH)
- **File**: `components/plugin-builder/RepoScanner.tsx` lines 144-148
- **Before**: `isSelected ? onRemoveSkill(key) : handleAddSkill(skill)` (used the old `key` variable)
- **After**: Constructs `skillKeyToRemove = \`repo:${url.trim()}:${ref.trim() || 'main'}:${skill.path}\`` explicitly before calling `onRemoveSkill`.
- **Root cause**: Even after the `key` variable is fixed, making the removal path explicit prevents any future drift between the lookup key and the removal key.

### Fix 4 — RepoScanner.tsx: `onSkillsFound` normalizes `ref` (MEDIUM)
- **File**: `components/plugin-builder/RepoScanner.tsx` line 52
- **Before**: `onSkillsFound(data.skills, url.trim(), ref)`
- **After**: `onSkillsFound(data.skills, url.trim(), ref.trim() || 'main')`
- **Root cause**: `ref` state could be an empty string if the user cleared the input; `handleAddSkill` already normalized to `'main'` but `onSkillsFound` passed the raw value, creating an inconsistency for the parent component.

## Verification

Re-read both files after edits confirmed all changes are present and syntactically correct. No other occurrences of the old key pattern exist in either file. The `core` and `marketplace` cases in `getSkillKey` are unaffected.

# Fix Report: FID00019 — activeOps counter not decremented when runBuild completes

## File Fixed
`services/plugin-builder-service.ts`

## Root Cause
`buildPlugin` fired `runBuild` as fire-and-forget and immediately returned a 202. The `finally` block in `buildPlugin` decremented `activeOps` when `buildPlugin` returned — which happened immediately after `runBuild` was launched, not when it finished. As a result, `activeOps` was always decremented prematurely (slot freed before the build completed), and if `runBuild` threw an unhandled rejection, no second decrement happened, meaning the counter became permanently stuck if the `.catch()` re-threw. However the actual production behavior was: the slot was freed too early (before the build finished), causing the concurrency guard to allow more builds than `MAX_CONCURRENT_OPS`.

## Fix Applied (lines ~379-413)

**Before:**
- `runBuild(...).catch(...)` — no `.finally()` on the promise chain
- `buildPlugin.finally()`: `if (opStarted) activeOps--` — always decremented when `buildPlugin` returned (immediately, not when build completed)

**After:**
- `runBuild(...).finally(() => { activeOps = Math.max(0, activeOps - 1) }).catch(...)` — decrements exactly when the async build actually finishes (success or failure)
- `buildPlugin.finally()`: `if (!opStarted) activeOps--` — only decrements on early error (before `runBuild` was ever launched), preventing double-decrement

## Invariants Preserved
- Early error (before `runBuild` starts): `buildPlugin.finally()` decrements once (`!opStarted` is true). `runBuild` never ran, so its `.finally()` never runs. Net: -1 exactly once.
- Normal path (`runBuild` launched): `buildPlugin.finally()` does NOT decrement (`!opStarted` is false). `runBuild.finally()` decrements when done. Net: -1 exactly once, at the correct time.
- No double-decrement possible.
- No slot leak on `runBuild` failure.

# TRDD-bb344037 — Harden the sudo token: bind to operation + subject, auth-before-consume, clamp, quota

**Source:** script↔API security audit, findings L2-M4, L2-L5, L2-L6, L2-L4 (`sudo-hardness` SUDO-01/02/03/04/05, `auth-layer-hardness` AUTH-HARD-04).
**Tier 2 (MANAGER):** core auth surface.

## Problem (WHY)

The sudo mechanism nails the hard parts — one-shot consume, 60 s TTL, 256-bit entropy, argon2-gated mint, and (critically) no CLI script can mint a token (scripts only pass through `AIMAESTRO_SUDO_TOKEN`). But four residual gaps degrade it from a per-action, per-principal human-presence factor to a fungible session-scoped one:

1. **Not bound to the operation (SUDO-01).** `SudoRecord` carries no route/method field; `requireSudoToken(req, method, path)` uses method+path only in the `requiresSudo()` predicate, then calls operation-agnostic `validateAndConsumeSudoToken(token)`. A token minted for op X is accepted on ANY of the 22 strict routes. (`lib/sudo-auth.ts:34-40,102-124`; `lib/sudo-guard.ts:28-41`)
2. **Subject minted but never checked (SUDO-02).** `issueSudoToken(password, subject)` stores `subject`; the validator RETURNS it; **zero consumers read it** (grep-verified). A token minted by principal A is accepted from B's request. (`app/api/auth/sudo-password/route.ts:53`; `lib/sudo-guard.ts:43-45`)
3. **Consume-before-auth on 13 of 22 strict routes (SUDO-04).** Those routes run `requireSudoToken` (one-shot consume) BEFORE `authenticateFromRequest`, so a shape-valid-but-fake-cookie caller can burn a token before failing auth → token-burn DoS. 9 routes already do auth-first correctly.
4. **TTL not re-clamped on config load (SUDO-03)** and **no per-subject mint quota (SUDO-05).**

Each is bounded today by the residual auth + RBAC under every strict route, so none is standalone escalation — but all are server-side and the binding data is already stored.

## Proposed change

```
1. lib/sudo-auth.ts: add `operation: {method, pathTemplate}` to SudoRecord; set it at issueSudoToken.
   lib/sudo-fetch.ts::mintSudoToken: accept + send the (method, pathTemplate) the token is for.
2. lib/sudo-guard.ts::requireSudoToken: after a valid consume,
   - resolve the consuming caller (buildAuthContext(authenticateFromRequest(request)) ->
     isSystemOwner ? 'system-owner' : agentId) and 403 if != result.subject;
   - 403 if result.operation != the consuming route's (method, pathTemplate).
3. Every strict route: run authenticateFromRequest (and reject) BEFORE requireSudoToken
   (flip the 13 sudo-first routes; the 9 auth-first routes already model it).
4. lib/security-config.ts::clampConfig: add clamp(sa,'sudoTokenTtlSeconds',10,600) and sessionTtlDays.
5. app/api/auth/sudo-password/route.ts: cap outstanding tokens per subject (<=2) and/or
   rate-limit per authenticated subject, not by the global op name.
```

## Acceptance criteria

- A token minted for `DELETE /api/agents/[id]` is rejected (403) on `POST /api/governance/password`.
- A token minted by agent A is rejected (403) when presented by agent B.
- An unauthenticated (fake-cookie) caller cannot consume/burn a token on any strict route (auth fails first).
- `sudoTokenTtlSeconds` from a tampered `.enc` is clamped to ≤600 on load.
- A subject cannot hold >2 outstanding tokens.
- Tests for each of the five, plus a regression that legitimate single-op sudo flows still work end-to-end.

## Risk / blast radius

Medium. The operation+subject binding changes the `mintSudoToken` call shape across the UI (`lib/sudo-fetch.ts`) — every minting call site must pass the target op. Roll the binding out behind the existing UI helper so callers don't each change. The auth-before-sudo reorder is mechanical but must be applied to all 13 routes consistently.

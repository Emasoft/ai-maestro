# TRDD-62e24f29 — Fix node-pty/Node-26 server crash-loop

## ⏵ STATE — READ THIS FIRST ON RESUME — 2026-06-21

**Found while prepping the #51 scenario verification (TRDD-903b7a20):** the pm2
`ai-maestro` instance is crash-looping on `ERR_DLOPEN_FAILED` from
`node_modules/node-pty/src/unixTerminal.ts:16` under **Node v26.3.0**. The
installed `node-pty@1.0.0` has no Node-26-ABI native binary, so `require('node-pty')`
fails → server crashes → pm2 restarts → loop. Terminal/PTY streaming (the
dashboard's core feature) is dead. The HTTP layer still answers (401 on
`/api/sessions`), but the fleet is NOT activation-ready.

**Ground truth (verified 2026-06-21T18:24):**
- `package.json` `node-pty: "^1.0.0"`, installed 1.0.0 (no `engines` field).
- `engines.node: ">=22.0.0"` (too loose — admits the broken Node 26); no `.nvmrc`.
- npm latest stable **node-pty 1.1.0** (+ 1.2.0-beta line) — newer than installed.
- Build toolchain present (python3, make, Xcode CLT) → node-gyp can build from source.

**OUTCOME (2026-06-21T18:30 — option 1 is the fix; option 2 is dead):**
- Option 2 (bump deps for Node 26) is NOT viable: `better-sqlite3@12.8.0` HARD-caps
  at Node 25 (`engines: 20.x||22.x||23.x||24.x||25.x`); `yarn install` under Node 26
  aborts on its engine check before anything builds. Bumping it to a Node-26 release
  is a risky DB-layer major bump — refused.
- The DLOPEN detail confirms ABI mismatch: native modules built for
  NODE_MODULE_VERSION 127 (Node 22), Node 26 needs 147.
- **`node_modules` is ALREADY correct for Node 22** (node-pty's binary is ABI-127),
  so NO reinstall is even needed — the sole fault is the machine's DEFAULT node became
  `26.3.0`, and pm2 launches under it.
- **PROVEN:** under the real `node@22` (`/opt/homebrew/opt/node@22/bin/node` →
  v22.22.3), BOTH deps load + work — `node-pty.spawn` is a function and a
  better-sqlite3 in-memory DB opens/exec/closes cleanly.
- **CORRUPTED-KEG GOTCHA (machine-specific):** the homebrew kegs `node@23/24/25/26`
  ALL report `v26.3.0` (mislabeled). Only `node@22` (v22.22.3) and nvm's `v20.5.1`
  are real ≤25 Nodes on this machine. So `.nvmrc` pins **22** (proven-working +
  the engines floor + LTS), not 24.

**In-repo fixes APPLIED (this is all that's in scope — `~/.pm2` + global Node are the
user's environment):**
1. `.nvmrc` → `22` (declare the project's Node; nvm/CI pick it up).
2. `package.json` `engines.node` `>=22.0.0` → `>=22.0.0 <26.0.0` (a Node-26 install now
   FAILS FAST with a clear "incompatible engine" message instead of crash-looping).
3. node-pty reverted to `^1.0.0` (the dep is fine; no bump needed).

**OPERATIONAL FIX — the user must apply (touches `~/.pm2` / global Node, outside the
project = out of autonomous scope):** point ai-maestro's pm2 at node@22 and restart:
```
pm2 delete ai-maestro
cd ~/ai-maestro && PATH="/opt/homebrew/opt/node@22/bin:$PATH" \
  pm2 start server.mjs --name ai-maestro --interpreter /opt/homebrew/opt/node@22/bin/node
```
(or `brew link --overwrite node@22` to make 22 the default, then `pm2 restart
ai-maestro --update-env`). Optionally `brew reinstall node@24` to repair the
corrupted kegs. After the switch, the server boots clean (no DLOPEN; PTY/terminal
streaming works) — which unblocks TRDD-903b7a20 #51.

**Reversibility:** package.json + yarn.lock are git-tracked (revert via
`git checkout`); node_modules is regeneratable. The pm2 instance is already
crash-looping, so a restart is strictly non-worse.

**NEXT ACTION:** in-repo fixes committed + pushed to fork. The running pm2 server stays
broken until the USER applies the operational fix above (point pm2 at node@22 — touches

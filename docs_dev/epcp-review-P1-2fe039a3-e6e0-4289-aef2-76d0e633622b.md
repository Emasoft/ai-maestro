### Issues Found

#### MUST-FIX

##### [SR-P1-001] Command injection in sendTmuxNotification via shell metacharacters
- **Severity:** MUST-FIX
- **Category:** security
- **Description:** The `sendTmuxNotification` function in `config-notification-service.ts` escapes double quotes but does not escape backticks, `$()`, `${}`, or other shell metacharacters. Both `sessionName` and `escaped` (derived from user-controllable agent names and operation strings) are interpolated into a shell command via `execAsync`. An agent name containing `$(malicious-command)` or backticks would be executed by the shell.
- **Evidence:** `services/config-notification-service.ts:107-110`
  ```typescript
  const escaped = truncated.replace(/"/g, '\\"')
  // ...
  await execAsync(`tmux display-message -t "${sessionName}" "[GOVERNANCE] ${escaped}"`, { timeout: 5000 })
  ```
  If `sessionName` = `foo$(rm -rf /tmp)`, the command becomes: `tmux display-message -t "foo$(rm -rf /tmp)" "..."` -- shell executes the subcommand.
- **Impact:** Remote code execution if an attacker can control agent names (via registration API) or governance request operation strings.
- **Recommendation:** Use `execFile` (no shell interpretation) instead of `exec`, passing arguments as an array: `execFileAsync('tmux', ['display-message', '-t', sessionName, \`[GOVERNANCE] ${truncated}\`], { timeout: 5000 })`. This eliminates shell metacharacter interpretation entirely.

##### [SR-P1-002] Governance request status shows 'executed' even when config deployment fails
- **Severity:** MUST-FIX
- **Category:** consistency
- **Description:** In `performRequestExecution` (cross-host-governance-service.ts), the governance request status is set to `'executed'` by `approveGovernanceRequest` BEFORE the actual deployment runs. If `deployConfigToAgent` returns an error, the function logs a warning and returns, but the status remains `'executed'`. This means the governance log, the UI, and synced peer hosts all show the request as successfully executed when the actual deployment failed. The governance sync broadcast at line 511 also propagates this false success.
- **Evidence:** `services/cross-host-governance-service.ts:287-288 and 493-495`
  ```typescript
  // Line 287-288: status already 'executed' before performRequestExecution runs
  if (updated.status === 'executed') {
    await performRequestExecution(updated)
  }

  // Line 493-495: deployment fails but status not reverted
  if (deployResult.error) {
    console.warn(`${LOG_PREFIX} configure-agent execution failed for request ${request.id}: ${deployResult.error}`)
    return  // returns without updating status back from 'executed'
  }
  ```
- **Impact:** Users and peer hosts see a config change as "executed" when it actually failed. This breaks trust in the governance audit trail.
- **Recommendation:** Either: (a) Add a `'failed'` status to `GovernanceRequestStatus` and update the request when deployment fails, or (b) separate the `approveGovernanceRequest` from `executeGovernanceRequest` so status only becomes `'executed'` after successful deployment.

#### SHOULD-FIX

##### [SR-P1-003] Config deploy endpoint in headless-router allows unauthenticated access (missing !auth.agentId guard)
- **Severity:** SHOULD-FIX
- **Category:** security
- **Description:** The `/api/agents/:id/config/deploy` route in `headless-router.ts` checks `auth.error` but does NOT check `if (!auth.agentId)`. When both `Authorization` and `X-Agent-Id` headers are absent, `authenticateAgent()` returns `{}` (no error, no agentId). The request passes the guard and reaches `deployConfigToAgent` with `auth.agentId = undefined`. This is inconsistent with other governance-gated routes (e.g., transfers at lines 1207-1211) that explicitly check `!auth.agentId` as a second gate.
- **Evidence:** `services/headless-router.ts:827-836` vs. `services/headless-router.ts:1207-1211`
  ```typescript
  // Config deploy route -- only checks error
  if (auth.error) { sendJson(res, 403, ...); return }
  // Missing: if (!auth.agentId) { sendJson(res, 401, ...); return }

  // Transfer route -- checks both
  if (auth.error) { sendJson(res, auth.status || 401, ...); return }
  if (!auth.agentId) { sendJson(res, 401, ...); return }
  ```
  Note: The same route also hardcodes `403` instead of `auth.status || 401` for auth errors, inconsistent with the rest of the file.
- **Impact:** Any unauthenticated HTTP request to the headless config deploy endpoint bypasses the auth guard. Because `deployedBy` is `undefined`, the downstream `deployConfigToAgent` will still execute (it only validates agent existence and operation, not who initiated it). The Next.js route has the same issue but since governance enforcement is opt-in (Phase 1), the practical impact is limited to headless mode where the endpoint is meant to be governance-gated.
- **Recommendation:** Add `if (!auth.agentId) { sendJson(res, 401, { error: 'Authenticated agent identity required for config deployment' }); return }` after the error check. Change `403` to `auth.status || 401` for consistency.

##### [SR-P1-004] canApprove role check is inverted: checks the viewed agent's role, not the viewer's role
- **Severity:** SHOULD-FIX
- **Category:** ux-concern
- **Description:** In `AgentSkillEditor.tsx`, `canApprove` is derived from `agentRole` which comes from `useGovernance(agentId)`. The `agentRole` computation checks whether `agentId` (the agent being VIEWED) is the manager or COS -- NOT whether the person currently using the UI is a manager or COS. This means:
  - When a manager views a regular agent's profile, `canApprove = false` (wrong: manager should be able to approve).
  - When anyone views the manager's profile, `canApprove = true` (wrong: viewing the manager's profile should not grant approval ability to all viewers).
- **Evidence:** `components/marketplace/AgentSkillEditor.tsx:70-72`
  ```tsx
  const { pendingConfigRequests, resolveConfigRequest, agentRole } = useGovernance(agentId)
  const agentPendingConfigs = pendingConfigRequests.filter(r => r.payload.agentId === agentId)
  const canApprove = agentRole === 'manager' || agentRole === 'chief-of-staff'
  ```
  `useGovernance.ts:46-54`: `agentRole` compares `agentId` against `managerId` and `chiefOfStaffId` -- it determines the role OF `agentId`, not the role of whoever is viewing.
- **Impact:** Approve/reject buttons appear in the wrong contexts. In a dashboard used by a single local user this is cosmetic (the server-side RBAC enforces actual authorization), but it creates a misleading UX that will confuse users and cause failed approval attempts.
- **Recommendation:** The UI needs the VIEWER's role, not the SUBJECT's role. Since AI Maestro is a localhost dashboard without user accounts, consider using the managerId directly: `canApprove = managerId !== null` (i.e., a password-protected system implies the local user has manager authority).

##### [SR-P1-005] updateSkills always uses 'add-skill' governance operation type regardless of actual operation
- **Severity:** SHOULD-FIX
- **Category:** logic
- **Description:** `updateSkills` in agents-skills-service.ts performs a single governance check with operation `'add-skill'` even when the caller only passes `body.remove` (skill removal) or `body.aiMaestro` (AI Maestro config update). This means the governance audit trail will incorrectly record an `add-skill` check for what was actually a `remove-skill` or `update-aimaestro-config` operation.
- **Evidence:** `services/agents-skills-service.ts:101`
  ```typescript
  const govCheck = checkConfigGovernance(agentId, requestingAgentId, 'add-skill')
  ```
- **Impact:** Audit trail inaccuracy. All operations currently have the same RBAC rules (COS or MANAGER), so no functional bug exists today. But if per-operation RBAC is added later, this will silently allow removals that should be denied.
- **Recommendation:** Determine the operation type from the body fields: use `'remove-skill'` when only `body.remove` is present, `'add-skill'` when `body.add` is present, etc.

##### [SR-P1-007] Approve/reject buttons have no loading state, no error feedback, no double-click protection
- **Severity:** SHOULD-FIX
- **Category:** ux-concern
- **Description:** The approve/reject buttons in `AgentSkillEditor.tsx` fire `resolveConfigRequest` immediately on click with no confirmation dialog, no loading/disabled state during the API call, and no error feedback if the call fails. The return value (which includes `success` and `error` fields) is not consumed. Users can double-click to fire duplicate requests, and failed approvals give no indication of failure.
- **Evidence:** `components/marketplace/AgentSkillEditor.tsx:295-306`
  ```tsx

  ```

- **Impact:** Poor UX. Double-clicks cause duplicate API calls. Failed approvals are silent, leaving users confused about whether their action succeeded.
- **Recommendation:** (1) Track an in-flight state per request ID (e.g., `Set<string>` of resolving request IDs) and disable buttons while pending. (2) Await the result and show an error toast/inline message on failure. (3) For reject, prompt for a reason string (the API supports it).

- **Description:** `deployBulkConfig` always calls `deployAddSkill` when `config.skills` is present and `deployAddPlugin` when `config.plugins` is present. ## 3. Risk Assessment

**Breaking changes:** The function signatures for `updateSkills`, `addSkill`, `removeSkill`, and `saveSkillSettings` have all gained a new `requestingAgentId` parameter. However, it defaults to `null`, so existing callers are not broken. The `GovernanceRequestPayload.configuration` type changed from `Record<string, unknown>` to `ConfigurationPayload` -- existing code that constructed this field with arbitrary objects will fail type checking. This is a breaking change for any external consumers of this type, but since this is a local application, the impact is contained.

**Data migration:** None needed. The new `ConfigurationPayload` type is used for new requests only. Existing governance requests in `governance-requests.json` with the old `configuration: Record<string, unknown>` shape will still be readable since `ConfigurationPayload` fields are all optional.

**Performance:** The `useGovernance` hook now makes 4 parallel fetch calls instead of 3 (added config requests endpoint). This is negligible. The duplicate hook instantiation (AgentProfile + AgentSkillEditor) doubles this to 8 calls but is a minor concern.

**Security:** Two security issues: (1) Command injection in `sendTmuxNotification` via unescaped shell metacharacters (MUST-FIX). (2) Missing `!auth.agentId` guard on the headless config deploy endpoint allows unauthenticated deployment (SHOULD-FIX). The Phase 1 backward compat pattern (null agentId bypasses all governance) is a known, documented design decision but means the RBAC system is advisory-only until auth headers are enforced.

## 5. Verdict Justification

This PR implements a significant and well-structured governance system for agent configuration operations. The service decomposition is clean, the RBAC enforcement is thorough, the cross-host governance flow is properly integrated, and the test coverage is extensive. The code quality is generally high with good attention to edge cases (path traversal, idempotency, TTL expiry, auto-rejection on deletion).

However, two issues require changes before merge:

1. **Command injection in sendTmuxNotification (SR-P1-001):** This is a real security vulnerability. Agent names and operation strings flow into a shell command with only double-quote escaping, leaving backtick and `$()` injection vectors open. The fix is straightforward (use `execFile` instead of `exec`).

2. **False 'executed' status on deployment failure (SR-P1-002):** The governance request status is set to 'executed' before the actual deployment runs, and never reverted on failure. This corrupts the governance audit trail and propagates false success to peer hosts. This is a design flaw that undermines trust in the governance system's accuracy.

The remaining SHOULD-FIX issues (inverted canApprove check, wrong operation types in governance checks, missing auth guard on headless endpoint, no UX feedback on approve/reject, asymmetric bulk-config) are important but not blocking. They represent UX bugs and logic inconsistencies that should be addressed soon after merge.

The risks of merging with just the two MUST-FIX items resolved are low: the RBAC system is advisory-only in Phase 1 (null agentId bypasses all checks), the type changes are backward-compatible, and the test coverage is solid for the core governance flow.

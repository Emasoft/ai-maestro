### Issues Found

#### MUST-FIX

##### [SR-P5-001] Governance enforcement bypassed in Full (Next.js) server mode for agent CRUD
- **Severity:** MUST-FIX
- **Category:** consistency / security
- **Description:** The headless router (`services/headless-router.ts:604-612, 965-981`) correctly calls `authenticateAgent()` and passes `requestingAgentId` to `createNewAgent`, `updateAgentById`, and `deleteAgentById` for Layer 5 governance enforcement. However, the Next.js route handlers (`app/api/agents/route.ts:54`, `app/api/agents/[id]/route.ts:35,57`) do NOT call `authenticateAgent()` and do NOT pass `requestingAgentId`. They call the service functions with only the required positional arguments, meaning `requestingAgentId` defaults to `undefined`, which is falsy, so the `if (requestingAgentId)` guard in the service layer is never triggered. This means **governance enforcement is completely inactive in full (Next.js) server mode** -- any unauthenticated HTTP request can create, update, or delete any agent.
- **Evidence:**
  ```typescript
  // app/api/agents/route.ts:54 -- NO auth, NO requestingAgentId
  const result = createNewAgent(body)

  // app/api/agents/[id]/route.ts:35 -- NO auth, NO requestingAgentId
  const result = updateAgentById(id, body)

  // app/api/agents/[id]/route.ts:57 -- NO auth, NO requestingAgentId
  const result = deleteAgentById(id, hard)

  // vs. headless-router.ts:604-612 -- HAS auth, PASSES requestingAgentId
  const auth = authenticateAgent(getHeader(req, 'Authorization'), getHeader(req, 'X-Agent-Id'))
  sendServiceResult(res, createNewAgent(body, auth.error ? null : auth.agentId))
  ```
  Other Next.js routes (teams, tasks, documents, transfers) DO call `authenticateAgent()`. Only the agent CRUD routes are missing it.
- **Impact:** In full server mode (the default `yarn dev` / `yarn start`), the entire Layer 5 governance system is a no-op for agent operations. An agent with `member` role can create/delete any agent via the web UI or direct HTTP call.
- **Recommendation:** Add `authenticateAgent()` calls to `app/api/agents/route.ts` (POST) and `app/api/agents/[id]/route.ts` (PATCH, DELETE), mirroring the headless-router pattern. Pass `auth.agentId` (or null on error) to the service functions.

##### [SR-P5-004] Governance enforcement is opt-in (silently bypassed when no auth headers present) -- by design, but undocumented
- **Severity:** SHOULD-FIX
- **Category:** design / ux-concern
- **Description:** The governance enforcement model is "progressive" -- if no auth headers are sent, the request is treated as "system owner / web UI" and governance checks are skipped (`agent-auth.ts:34`, `agents-core-service.ts:604`). This means governance only applies when agents authenticate themselves. This is a deliberate backward-compatibility choice (documented in `agent-auth.ts` comments), but it has a subtle UX consequence: **a user who sets up governance, assigns a manager, and creates closed teams will still see that any HTTP client without auth headers can bypass all restrictions.** There is no way to make governance mandatory (i.e., "require auth for all mutations").
- **Evidence:**
  ```typescript
  // agent-auth.ts:31-36
  if (authHeader === null && agentIdHeader === null) {
    return {}  // system owner -- governance not enforced
  }
  ```
- **Impact:** Users who expect governance to restrict ALL access (not just authenticated agent access) will be surprised. This is a design judgment call, not a bug, but it should be clearly documented.
- **Recommendation:** Add a configuration option (e.g., `requireAuthForMutations: boolean` in governance config) to allow operators to lock down the system. At minimum, document this behavior prominently in the CHANGELOG or README.

##### [SR-P5-006] Cross-host COS attestation recipientHostId binding not verified by receiver
- **Severity:** SHOULD-FIX
- **Category:** security
- **Description:** This was already found by CC-P5-A2-002 (governance-core correctness report), but I want to elevate it from a per-file finding to a holistic concern: the `HostAttestation` type has a `recipientHostId` field specifically documented as preventing "cross-target replay" attacks, but the verification function never checks it. This means a signed attestation intended for Host A can be replayed against Host B. The signed data includes `recipientHostId` (so it is covered by the signature), but the verifier does not compare `attestation.recipientHostId` against its own host ID.
- **Impact:** In a multi-host mesh, a compromised host could capture attestations and replay them against other hosts.
- **Recommendation:** Add `recipientHostId` validation in `verifyRoleAttestation()` -- compare `attestation.recipientHostId` against `getSelfHostId()` when the field is present.

**Data migration:**
- `loadTeams()` migration guarantees `type` field is populated on load (defaults to `'open'`). This is safe for existing data. Risk: LOW.
- New files created on disk: `~/.aimaestro/governance.json`, `~/.aimaestro/governance-requests.json`, `~/.aimaestro/host-keys/`, `~/.aimaestro/manager-trust.json`, `~/.aimaestro/governance-peers/`. All initialized lazily on first access with safe defaults. Risk: LOW.

**Performance:**
- `checkMessageAllowed()` loads teams and governance from disk on every call (synchronous `readFileSync`). In a high-message-throughput scenario, this could be a bottleneck. The existing `loadTeams()` has its own caching, but `loadGovernance()` reads from disk each time. Risk: MEDIUM (for high-throughput deployments).
- Governance peer cache has TTL-based eviction (300s default). Risk: LOW.

**Security:**
- SR-P5-001 (governance bypass in full mode) is the most significant security concern.
- SR-P5-006 (recipientHostId not verified) enables attestation replay across hosts.
- CozoDB injection (CC-P5-A5-001 through -003) and shell injection (CC-P5-A5-004) were found by the correctness swarm and are real security issues.
- Unsalted SHA-256 for API keys (CC-P5-A4-007) is acceptable given 256-bit key entropy but below industry best practice.

## 5. Verdict Justification

**Verdict: REQUEST CHANGES**

The governance system is well-designed at the type and algorithm level. The message filter, transfer resolve, and attestation code show production-quality thinking with proper locking, compensating actions, and edge case handling. The test coverage is substantial (171 tests) and meaningful.

However, there are two blocking issues:

1. **SR-P5-001: Governance is completely bypassed in Full (Next.js) mode** for agent CRUD operations. This is not a minor inconsistency -- it means the entire Layer 5 enforcement system is a no-op in the default server mode. The headless router has the auth calls, but the Next.js routes do not. This misrepresents the feature's completeness. Either implement it or remove the claim.

The overall architecture is sound. The 4-layer governance model (peer cache, attestation, cross-host requests, manager trust) is well-structured. After fixing the two MUST-FIX issues, this would be ready for merge with SHOULD-FIX items tracked as follow-up issues.

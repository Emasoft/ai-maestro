## 2. Code Quality

### Strengths

**A: Service layer architecture and ServiceResult pattern**
The governance system follows a clean service layer pattern. All API routes (both Next.js and headless) are thin wrappers that call service functions returning `ServiceResult<T>`. The shared `ServiceResult<T>` type from `types/service.ts` replaces 15+ duplicate definitions. This is disciplined architecture.

**A: Message filter design (lib/message-filter.ts)**
The 189-line message filter is one of the best-designed files in this PR. The algorithm is clearly documented (R6.1-R6.7), the code follows the documented steps exactly, and edge cases are handled thoughtfully. The single-snapshot pattern (loading teams and governance once) avoids redundant file reads. The `isValidUuid` check on recipientAgentId prevents alias-based governance bypass.

**A-: Input validation consistency (API routes)**
The Next.js API routes for governance requests validate every query parameter against known value sets (`VALID_GOVERNANCE_REQUEST_STATUSES`, `VALID_GOVERNANCE_REQUEST_TYPES`, `VALID_REQUESTED_BY_ROLES`) and format-check UUIDs and hostnames. The POST body validation is thorough with explicit field-by-field checks.

**A-: Team validation (lib/team-registry.ts:validateTeamMutation)**
The centralized validation function enforces a comprehensive set of business rules (R1-R4) in one place, called by both createTeam and updateTeam. The auto-downgrade from closed to open when COS is removed (G5) is elegant. The multi-closed-team constraint correctly exempts MANAGER but not COS.

**B+: Locking and atomic writes**
File-based state uses `withLock` for all mutations and temp-file + `renameSync` for atomic persistence. The `acquireLock` pattern with 30-second timeout prevents deadlocks. Transfer resolution uses a compensating action (revert to pending) if team save fails after approval.

**B+: Rate limiting with TOCTOU mitigation**
Cross-host governance uses `checkAndRecordAttempt` (atomic check-and-record) instead of separate `checkRateLimit` + `recordFailure` calls, eliminating the TOCTOU window. Per-agent rate limit keys prevent one agent's failures from locking out others.

**B+: Security hardening**
Ed25519 signatures on all cross-host governance messages. Timestamp validation with 5-minute window (allowing 1-minute clock skew). Forced `status: 'pending'` and cleared approvals on received requests prevent malicious peers from pre-filling approvals.

**B: Test coverage (780 tests, all passing)**
28 test files with 780 tests covering all governance layers. Key areas: governance core, message filter, transfer registry, cross-host governance, governance request registry, manager trust, host keys, role attestation, agent config governance. Tests use proper mocking with fs store patterns.

### Issues Found

#### SHOULD-FIX

##### [SR-P6-001] Headless router GET governance requests lacks input validation that Next.js route has
- **Severity:** SHOULD-FIX
- **Category:** consistency
- **Description:** The Next.js route `app/api/v1/governance/requests/route.ts:126-170` validates `status` against `VALID_GOVERNANCE_REQUEST_STATUSES`, `type` against `VALID_GOVERNANCE_REQUEST_TYPES`, `agentId` format via `isValidUuid`, and `hostId` via `HOSTNAME_RE`. The headless router at `services/headless-router.ts:1422-1429` passes all four params through without any validation. An attacker sending `?status=foo` in headless mode gets unsanitized strings cast to typed parameters. In practice, `listCrossHostRequests` just does string comparison filtering, so the impact is limited to confusing no-results responses rather than crashes. But it violates the principle of consistent validation across server modes.
- **Evidence:** `services/headless-router.ts:1422-1429` -- no validation on query params; `app/api/v1/governance/requests/route.ts:130-154` -- full validation with error messages.
- **Impact:** In headless mode, invalid query parameter values are silently accepted. No crash risk, but inconsistent behavior between server modes.
- **Recommendation:** Extract the validation constants (`VALID_GOVERNANCE_REQUEST_STATUSES`, `VALID_GOVERNANCE_REQUEST_TYPES`, `HOSTNAME_RE`) to a shared module and apply them in both handlers.

##### [SR-P6-002] Headless router POST governance requests (local path) lacks body validation that Next.js route has
- **Severity:** SHOULD-FIX
- **Category:** consistency
- **Description:** The Next.js route `app/api/v1/governance/requests/route.ts:68-93` validates 6 body fields (type, password, targetHostId, requestedBy, requestedByRole, payload.agentId) before calling `submitCrossHostRequest`. The headless router at `services/headless-router.ts:1417-1420` passes the raw body directly. The downstream `submitCrossHostRequest` does validate most fields, but not all (e.g., `requestedByRole` validation against known values happens only in the Next.js route layer).
- **Evidence:** `services/headless-router.ts:1417-1420` -- `sendServiceResult(res, await submitCrossHostRequest(body))` with no body validation; `app/api/v1/governance/requests/route.ts:68-93` -- extensive field-by-field validation.
- **Impact:** Headless mode accepts malformed requests that the service layer may or may not catch. The `requestedByRole` validation is only in the Next.js route, so headless mode can receive arbitrary strings for this field.
- **Recommendation:** Mirror the Next.js body validation in the headless handler, or (better) move all validation into the service layer so both handlers share it.

##### [SR-P6-003] `performed-but-failed` execution state is not tracked
- **Severity:** SHOULD-FIX
- **Category:** design
- **Description:** In `services/cross-host-governance-service.ts:404-406`, the code comments explicitly acknowledge that `performRequestExecution` can fail after the request status has already been set to `'executed'`. The `'executed'` status means "execution was attempted" not "succeeded". If `saveTeams()` fails in `add-to-team`, `remove-from-team`, `assign-cos`, or `remove-cos`, or if `deployConfigToAgent` returns an error for `configure-agent`, the failure is only logged -- the request status remains `'executed'`. The comment says a proper `'failed'` status requires changes to the type and all callers, deferred to Phase 2.
- **Evidence:** `services/cross-host-governance-service.ts:529-533` -- configure-agent deploy error only logged; `services/cross-host-governance-service.ts:404-406` -- comment explaining the limitation.
- **Impact:** Operators cannot distinguish successful from failed executions by looking at the request registry. A failed deploy silently appears as "executed." This is a known and documented limitation.
- **Recommendation:** At minimum, add an optional `executionError?: string` field to `GovernanceRequest` that `performRequestExecution` populates on failure. This doesn't require a new status value but does let callers see that execution failed.

For a localhost dashboard with single-digit concurrent users, this is acceptable. The code consistently marks these as "TODO Phase 2: Add in-memory caching." Timestamp validation on signatures. Forced-pending status on received requests. UUID validation on path parameters. Overall security posture is solid for a Phase 1 localhost system.

- Layer 1: State replication via governance-peers.ts and governance-sync.ts
- Layer 2: Cryptographic identity attestation via Ed25519 host keys and signed role attestations
- Layer 3: Cross-host governance requests with a proper state machine (pending -> remote-approved/local-approved/dual-approved -> executed/rejected)
- Layer 4: Manager trust registry for auto-approval
- Layer 5: Agent configuration governance with RBAC enforcement

Each layer builds on the previous ones with clear separation of concerns.

The discriminated union types (GovernanceSyncType, GovernanceRequestType, GovernanceRequestStatus) correctly model the state machines.

**Security hardening (A).** Rate limiting on password verification, bcrypt with 12 salt rounds, 72-char bcrypt limit enforcement, UUID validation on path parameters, JSON parse guards, Ed25519 signature verification for cross-host messages, timestamp freshness checks with skew tolerance. These are consistently applied across all governance endpoints.

**Test coverage (A-).** 867 tests passing across 30 test files. Tests use real crypto verification (not mocked) where appropriate.

**Error handling (A).** ServiceResult<T> pattern is consistently used across all service functions. Error messages are specific and actionable (e.g., "Governance password not set. RoleAssignmentDialog handles complex role transitions (manager -> COS -> member and back) with proper cleanup on partial failure. ### Issues Found

#### MUST-FIX

#### SHOULD-FIX

##### [SR-P7-001] Governance API routes bypass service layer (duplicated business logic)
- **Severity:** SHOULD-FIX
- **Category:** design
- **Description:** The governance API routes `/api/governance/manager/route.ts` and `/api/governance/password/route.ts` implement their business logic directly (importing from `lib/governance`, `lib/rate-limit`, `lib/agent-registry`) instead of delegating to the service functions in `services/governance-service.ts` (`setManagerRole`, `setGovernancePassword`). Meanwhile, the headless router correctly delegates to the service layer. This creates a dual-maintenance burden -- any bug fix to the business logic must be applied in two places, and the implementations can drift.
- **Evidence:**
  - `app/api/governance/password/route.ts:2` imports directly from `@/lib/governance`
  - `app/api/governance/manager/route.ts:2` imports directly from `@/lib/governance`
  - `services/governance-service.ts:105-147` contains `setGovernancePassword` with identical logic
  - `services/governance-service.ts:57-100` contains `setManagerRole` with identical logic
  - `services/headless-router.ts:1234` correctly delegates: `sendServiceResult(res, await setGovernancePassword(body))`
  - Compare with trust routes which DO delegate: `app/api/governance/trust/route.ts:12` imports from `@/services/governance-service`
- **Impact:** Future bugs fixed in one copy but not the other. Different behavior between Next.js and headless modes.
- **Recommendation:** Refactor the password and manager routes to delegate to their service functions, matching the pattern used by the trust routes and all other routes in the codebase.

##### [SR-P7-002] CHANGELOG test count is stale (claims 169, actual is 199)

- **Category:** consistency
- **Description:** The CHANGELOG for v0.26.0 states "169 new tests across 9 test files" but the actual count across those 9 files is 199. The count was likely accurate when the CHANGELOG entry was written but was not updated after subsequent fix passes added more tests.

##### [SR-P7-004] GovernancePasswordDialog has no maximum attempt feedback beyond rate limiting
- **Severity:** NIT
- **Category:** ux-concern
- **Description:** When a user enters the wrong governance password in confirm mode, the error comes from the server-side `onPasswordConfirmed` callback. If the server returns a rate-limit error (429), the dialog shows the generic error message but does not visually distinguish between "wrong password" and "too many attempts -- locked out". The user might keep retrying without understanding they are locked out.
- **Evidence:** `components/governance/GovernancePasswordDialog.tsx:104` -- the catch block uses `e.message` which would contain the rate limit message, but there's no special visual treatment (e.g., a warning icon, different color, countdown timer).
- **Impact:** Minor UX confusion when rate-limited. The error message is still displayed, so the user gets the information -- it's just not visually prominent.
- **Recommendation:** Consider adding a visual distinction for rate-limit errors (e.g., a yellow warning banner instead of red error text, or a countdown timer showing when they can retry).

## 3. Risk Assessment

**Breaking changes:** One notable breaking change: the role rename from `'normal'` to `'member'`. This is documented in the CHANGELOG and has been consistently applied across the entire codebase. No references to the old `'normal'` role name remain in any TypeScript file. However, any external consumers (CLI scripts, AMP plugins) that use the old role name would break. The `AgentRole` type at `types/agent.ts:435` is the single source of truth. **Risk: LOW** -- the old value was internal and Phase 1 is localhost-only.

**Data migration:** The `loadTeams()` function includes a migration path that populates the now-required `type` field (defaulting to `'open'`) on teams that were created before v0.26.0. The `loadGovernance()` function creates `governance.json` with defaults if it does not exist. Both migration paths are safe -- they add new data without removing existing data. **Risk: LOW**.

**Performance:** The governance library re-reads `governance.json` from disk on every call (`isManager`, `isChiefOfStaff`, etc.). This is acceptable for Phase 1 localhost but would need caching for production multi-host deployments. The code includes TODO comments acknowledging this. **Risk: LOW** for current phase.

**Security:** The governance password is properly hashed with bcrypt (12 rounds), rate-limited, and validates the 72-char bcrypt input limit. Ed25519 signatures are used for cross-host attestation with timestamp freshness checks. The only security concern already flagged is the CozoDB injection in `agents-graph-service.ts` (CC-P7-A4-001). **Risk: LOW** after that fix.

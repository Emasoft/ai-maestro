### [SF-001] Unsafe cast of `payload.context` from `unknown` to `Record<string, unknown>`
**File:** /Users/emanuelesabetta/ai-maestro/lib/messageQueue.ts:278
**Original IDs:** CC-P10-A1-001
**Phases:** CC
**Scope:** IN-SCOPE (governance build fix in messageQueue.ts)

The `payload.context` field is typed as `unknown` in the `AMPEnvelopeMsg` interface (line 218). At line 278, it is cast directly to `Record<string, unknown> | undefined` without any runtime validation that it is actually an object. If a malformed AMP message file contains `context: "some string"`, `context: 42`, or `context: [1,2,3]`, the cast will succeed silently but downstream code expecting an object could produce incorrect results or runtime errors.

The `status` field has proper validation (lines 250-253 with `validStatuses.includes()`), but `context` has no type-shape validation.

**Fix:** Add a runtime check before casting:
```typescript
context: (payload.context && typeof payload.context === 'object' && !Array.isArray(payload.context))
  ? payload.context as Record<string, unknown>
  : undefined,
```

### [NT-001] Non-null assertions on `result.data!` rely on implicit convention, not type narrowing
**File:** /Users/emanuelesabetta/ai-maestro/app/api/v1/health/route.ts:19, info/route.ts:19, messages/pending/route.ts:28,43,63, register/route.ts:28, route/route.ts:37
**Original IDs:** CC-P10-A1-002
**Phases:** CC
**Scope:** IN-SCOPE (API route non-null assertions)

All 7 uses of `result.data!` across the 5 API route files follow the pattern: `if (result.error) return ...` followed by `result.data!`. The `ServiceResult<T>` type defines both `data?: T` and `error?: string` as optional, so TypeScript cannot narrow `data` to `T` after checking `error` is falsy. The non-null assertions are **safe at runtime** (verified all service function paths), but the pattern is fragile.

A better pattern would be a discriminated union for `ServiceResult`:
```typescript
type ServiceResult<T> =
  | { data: T; status: number; headers?: Record<string, string>; error?: never }
  | { error: string; status: number; headers?: Record<string, string>; data?: never }
```

**Fix:** Consider refactoring `ServiceResult` to a discriminated union. Low priority since all current usages are verified safe.

### [NT-002] Priority cast relies on `||` fallback but lacks validation like `status` does
**File:** /Users/emanuelesabetta/ai-maestro/lib/messageQueue.ts:273
**Original IDs:** CC-P10-A1-003
**Phases:** CC
**Scope:** IN-SCOPE (governance build fix in messageQueue.ts)

The `priority` field (line 273) uses `(envelope.priority || 'normal')` as a fallback and casts directly to the union type. Compare with the `status` field (lines 250-253) which has explicit validation with `validStatuses.includes()`. If `envelope.priority` contains an unexpected string, the cast would pass a value that doesn't match any union members.

Similarly, `content.type` at line 276 uses `(payload.type || 'notification')` and casts directly without validation.

**Fix:** Apply the same validation pattern used for `status`:
```typescript
const validPriorities: Message['priority'][] = ['low', 'normal', 'high', 'urgent']
const rawPriority = envelope.priority || 'normal'
const priority: Message['priority'] = validPriorities.includes(rawPriority as Message['priority'])
  ? (rawPriority as Message['priority']) : 'normal'
```

### [MF-001] `activeOps` double-decrement race condition in `buildPlugin` (UPSTREAM)
**File:** /Users/emanuelesabetta/ai-maestro/services/plugin-builder-service.ts:307-388
**Original IDs:** CC-P10-A0-002
**Phases:** CC
**Scope:** UPSTREAM -- not in scope for this PR

In `buildPlugin()`, when the async `runBuild` promise is launched at line 367, its `.finally()` handler decrements `activeOps`. However, if the outer try block throws after `runBuild` is launched but before it resolves, the catch block also decrements `activeOps`, creating a double-decrement. In practice the risk is very low (no throwing code exists between the runBuild call and the return), but the pattern is structurally fragile.

**Fix:** Move the `activeOps++` to just before the `runBuild` call, and remove the decrement from the outer catch block.

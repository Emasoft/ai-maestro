### [SF-002] Hostname regex rejects single-character hostnames
**File:** app/api/v1/governance/requests/route.ts:129
**Original IDs:** CC-P4-A1-002
**Phases:** CC
The `HOSTNAME_RE` regex `/^[a-zA-Z0-9][a-zA-Z0-9._-]{0,253}[a-zA-Z0-9]$/` requires a minimum of 2 characters. Single-character hostnames (e.g., "a") are valid per RFC 952/1123 but would be rejected. Also, the regex allows hostnames up to 255 characters (1 + 253 + 1) which exceeds RFC 1035's 253-octet limit for FQDNs.

**Fix:** Change to `/^[a-zA-Z0-9]([a-zA-Z0-9._-]{0,251}[a-zA-Z0-9])?$/` to allow single-char hostnames and correctly cap at 253 chars total.

### [SF-003] TOCTOU race: pre-check for terminal state is outside the lock, enabling double-execution
**File:** services/cross-host-governance-service.ts:293-305
**Original IDs:** CC-P4-A2-001
**Phases:** CC
In `approveCrossHostRequest`, the terminal-state pre-check at line 293 reads the request status at line 265 (lock-free via `getGovernanceRequest`), but the actual approval at line 298 (`approveGovernanceRequest`) acquires the `governance-requests` lock and re-reads. Between lines 265 and 298, a concurrent request could have already transitioned the request to `executed`. The inner `approveGovernanceRequest` returns the request (not null) for terminal states, so the caller sees `updated.status === 'executed'` and calls `performRequestExecution` again. A `configure-agent` execution could double-deploy a configuration and send duplicate notifications.

**Fix:** After `approveGovernanceRequest` returns, check whether the approval actually changed the status. Have `approveGovernanceRequest` return a richer result (e.g., `{ request, wasAlreadyTerminal: boolean }`), or compare the pre-read status against the post-approval status to determine if the transition was caused by this call.

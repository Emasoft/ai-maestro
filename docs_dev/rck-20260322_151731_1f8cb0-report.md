# Fix Report: scan-repo/route.ts

**Date**: 2026-03-22
**File**: `app/api/plugin-builder/scan-repo/route.ts`
**Finding**: BUG — Missing type validation for `body.ref` (severity: medium)

## Root Cause

Line 22 (original):
```typescript
const result = await scanRepo(body.url, body.ref || 'main')
```

`body.ref` is parsed from untrusted JSON. If a caller sends `body.ref` as a non-string (e.g., a number, array, or object), the `||` fallback only triggers for falsy values — a non-empty number or object would be passed directly to `scanRepo(url: string, ref: string)`, violating its type contract.

## Fix Applied

Replaced the single-expression fallback with an explicit guard:

1. If `body.ref` is present but not a string → return HTTP 400 immediately.
2. Otherwise assign `ref` as `body.ref` (string) or `'main'` (default).
3. Pass the validated `ref` variable to `scanRepo`.

```typescript
// Validate ref is a string when provided — scanRepo expects string
if (body.ref !== undefined && typeof body.ref !== 'string') {
  return NextResponse.json(
    { error: 'Repository reference (ref) must be a string' },
    { status: 400 }
  )
}
const ref: string = typeof body.ref === 'string' ? body.ref : 'main'

const result = await scanRepo(body.url, ref)
```

## Verification

- No other callers of `scanRepo` in this file.
- `scanRepo` signature confirmed: `scanRepo(url: string, ref: string = 'main')`.
- The existing `body.url` string validation pattern is now consistently applied to `body.ref`.
- All existing error-handling branches (`SyntaxError`, generic 500) are untouched.

- **Location**: line 186 (JSX, "Push to GitHub" button `disabled` prop)

- **Now**: `disabled={!isComplete}`
- **Root cause**: The button opens the push panel so the user can enter the URL. - **Location**: `handleBuild` — two call sites: inside polling interval on `statusData.status !== 'building'`, and in the synchronous `else` branch when initial status is already complete
- **Was**: Both branches called `setShowLogs(true)` on completion
- **Now**: Both `setShowLogs(true)` calls removed; logs remain collapsed by default and expand only when the user clicks the toggle
- **Root cause**: Forcing logs open on every build completion overrides the user's explicit preference for collapsed logs, which is the initial state. The user controls log visibility via the chevron toggle button.

### Fix 3 (medium): JSON parse error in polling interval leaves component stuck in `building` state
- **Location**: `handleBuild` → `setInterval` callback, `statusRes.json()` call

- **Location**: `handlePush`, client-side URL validation regex
- **Was**: `/^https:\/\/github\.com\/.+\/.+/` — uses `.+` which is too broad (matches any character including `/`, so `https://github.com/a/b/c/d` would pass), and does not allow the common `.git` suffix
- **Now**: `/^https:\/\/github\.com\/[a-zA-Z0-9._-]+\/[a-zA-Z0-9._-]+(\.git)?$/` — explicitly allows only valid GitHub owner/repo name characters (alphanumeric, dot, underscore, hyphen), optionally followed by `.git`, anchored at end so no extra path segments are accepted

# Fix Report: BuildAction setPushing stuck in loading state

**File**: `components/plugin-builder/BuildAction.tsx`
**Timestamp**: 2026-03-22T15:12:00Z
**Finding**: rck-20260322_150917_1f8cb0-[LP00002-IT00004-FID00008]

## Root Cause

In `handlePush`, the URL validation block appeared before `setPushing(true)` in the original code — meaning the button never entered loading state before validation ran. This is the actual ordering bug: `setPushing(true)` should fire first (so the button enters loading state immediately on click), then validation runs, and the validation failure branch must call `setPushing(false)` before returning, or the button is permanently stuck in loading state.

## Fix Applied

Moved `setPushing(true)` and `setPushResult(null)` to BEFORE the URL validation block, and added `setPushing(false)` in the validation failure branch before `return`.

**Before** (lines 129-141, original):
```
handlePush:
  if (!forkUrl.trim() || !result?.manifest) return         // early exit (ok — before setPushing)
  // URL validation block (no setPushing yet)
  if (!url.match(...)) {
    setPushResult({ ok: false, ... })
    return                                                  // BUG: pushing never set, but ordering wrong
  }
  setPushing(true)    // too late — should be first
  setPushResult(null)
```

**After** (lines 129-141, fixed):
```
handlePush:
  if (!forkUrl.trim() || !result?.manifest) return         // early exit (ok — pushing not yet true)
  setPushing(true)                                         // enters loading state immediately
  setPushResult(null)
  if (!url.match(...)) {
    setPushResult({ ok: false, ... })
    setPushing(false)                                      // FIXED: resets loading state on validation fail
    return
  }
  // ... fetch ...
  finally { setPushing(false) }                            // also resets on success/error
```

## Verification

Read the final state of `handlePush` (lines 129-163) — the function now correctly:
1. Returns immediately (before `setPushing(true)`) if no URL or no manifest.
2. Calls `setPushing(true)` as the first real action.
3. Calls `setPushing(false)` on URL validation failure before returning.
4. Calls `setPushing(false)` via `finally` on fetch success or error.

No other occurrences of this pattern exist elsewhere in the file.

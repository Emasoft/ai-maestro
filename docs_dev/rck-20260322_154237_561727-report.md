## Finding 1 (HIGH): portable_sed regex for safe_dir — backslash escaping

**Status**: FIXED

**Location**: `act4_start_and_register`, line 1177

**Root cause**: The sed character class `[[\.*^$|&\\\/]` was malformed. In a POSIX character class, `]` terminates the class unless it appears as the very first character. Placing `[` at position 0 followed by `\` and then `]` caused the class to be parsed as `[[` (opening) then `\.*^$|&\\\/` with the `]` closing the class immediately after the first `[` — meaning `]` itself was never included in the escape set, and the second `[` closed the class early, breaking the regex structure. Backslash and bracket characters in INSTALL_DIR paths would not be correctly escaped before being passed to `portable_sed`.

**Fix applied**:
```bash
# Before (buggy — ] not escaped, class structure malformed):
safe_dir=$(printf '%s' "$INSTALL_DIR" | sed 's/[[\.*^$|&\\\/]/\\&/g')

# After (correct — ] first in class, [ anywhere inside, \\ before /):
safe_dir=$(printf '%s' "$INSTALL_DIR" | sed 's/[][\.*^$|&\\/]/\\&/g')
```

The correct POSIX idiom to include `]` in a character class is to place it as the first character immediately after `[` (or `[^` for negation). This means `[][...]` is the portable way to include both `]` and `[` in the same class. The fix uses `[][\.*^$|&\\/]` which correctly escapes: `]`, `[`, `.`, `\`, `*`, `^`, `$`, `|`, `&`, `/`.

## Analysis

### "Inconsistent activeOps decrement" (HIGH) — FALSE POSITIVE

The reviewer flags the `catch` block decrement of `activeOps` in `buildPlugin` as a bug.
This is incorrect. The code is logically sound:

1. `activeOps++` happens at the top of the `try` block.
2. If any setup step (`fs.mkdir`, `fs.writeFile`, etc.) throws synchronously, the `catch` block is hit.
   At this point, `runBuild` was NEVER called → `runBuild`'s `finally` will NEVER run → the `catch` decrement is the only place to decrement → CORRECT.
3. If all setup succeeds: `runBuild(...)` is called synchronously (returns a Promise), 
   then `return { data: result, status: 202 }` executes → the `try` block exits normally → `catch` does NOT fire.
   Later, `runBuild`'s `finally` decrements `activeOps` → CORRECT.

The `catch` block can ONLY fire when `runBuild` was NOT called. There is no double-decrement path.
Removing the `catch` decrement would cause a resource leak when setup fails.

**Conclusion**: Current implementation is correct. No fix applied.

BUG: Incomplete Error Handling for `pushToGitHub` Service Result (Medium)

The `ServiceResult<T>` interface defined in `services/marketplace-service.ts` used all-optional
fields (`data?: T`, `error?: string`), which allowed TypeScript to accept a value `{ status: N }`
carrying neither field — a state that would cause `NextResponse.json(undefined)` in any route

## Verification

  `services/plugin-builder-service.ts` (the only TypeScript consumer of that import).
- Confirmed all return sites in `plugin-builder-service.ts` (`buildPlugin`, `getBuildStatus`,
  `scanRepo`, `pushToGitHub`) and `marketplace-service.ts` itself always return either `data` or
  `error` — never both, never neither. The invalid state was a type-system gap, not a runtime path.
- The existing guard in `route.ts` (`if (!result.data)` after `if (result.error)`) is a correct
  runtime safeguard; it remains in place unchanged.

## Fix Applied

**File changed:** `services/marketplace-service.ts`

Replaced the loose `interface ServiceResult<T>` (all-optional fields) with a proper discriminated
union type that enforces at the TypeScript level that exactly one of `data` (success branch) or
`error` (failure branch) is always present:

```typescript
// Before
export interface ServiceResult<T> {
  data?: T
  error?: string
  status: number
}

// After
```

- A success result `{ data: T, status: N }` is valid; adding `error` is a compile error.
- An error result `{ error: string, status: N }` is valid; adding `data` is a compile error.
- A bare `{ status: N }` with neither is a compile error — the invalid state can no longer be
  constructed silently.

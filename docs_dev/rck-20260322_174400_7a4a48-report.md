## app/api/plugin-builder/build/route.ts (FID00002)

# Fix Report — LP00003-IT00001-FID00002

**File:** `app/api/plugin-builder/build/route.ts`
**Issue:** Generic catch block masked service-layer errors as 400 "Invalid request body"

## Root Cause

The single `try/catch` in the `POST` handler wrapped both `request.json()` and `buildPlugin(body)`. Any unexpected exception thrown inside `buildPlugin` (e.g., import failure, unhandled service error) was silently caught and returned to the client as `{ error: 'Invalid request body', status: 400 }`. This is semantically incorrect — a 400 response means the client sent a bad request, but in these cases the error originates in the server's service layer.

## Fix Applied

Separated the JSON parse into its own `try/catch` that returns 400 only for malformed JSON. The call to `buildPlugin(body)` is moved outside the catch block so unexpected service errors propagate naturally (resulting in a 500 from Next.js's default error handler rather than a misleading 400).

**Before:**
```typescript
export async function POST(request: NextRequest) {
  try {
    const body = await request.json()
    const result = await buildPlugin(body)
    if (result.error) {
      return NextResponse.json({ error: result.error }, { status: result.status })
    }
    return NextResponse.json(result.data, { status: result.status })
  } catch {
    return NextResponse.json({ error: 'Invalid request body' }, { status: 400 })
  }
}
```

**After:**
```typescript
export async function POST(request: NextRequest) {
  // Only catch JSON parse errors here — service errors must propagate as 500
  let body: unknown
  try {
    body = await request.json()
  } catch {
    return NextResponse.json({ error: 'Invalid request body' }, { status: 400 })
  }

  const result = await buildPlugin(body)

  if (result.error) {
    return NextResponse.json({ error: result.error }, { status: result.status })
  }
  return NextResponse.json(result.data, { status: result.status })
}
```

## Verification

- The `try/catch` now exclusively guards `request.json()` — only malformed JSON bodies return 400.
- `buildPlugin(body)` is called outside the catch block; any unhandled exception it throws will propagate to Next.js's top-level error boundary, resulting in a 500 response with no misleading error message.
- The existing `result.error` handling path is preserved unchanged — known service errors still return the correct status code supplied by the service layer.
- No other files were affected; no callers of this route rely on the catch path returning 400 for non-JSON errors (that was the bug being fixed).

## Status

DONE — fix applied cleanly, file verified.

## Issue
`getSkillKey` for repo type omitted `ref` from the key, causing the "already selected" indicator in `RepoScanner.tsx` to never trigger.

## Root Cause
`getSkillKey` built repo keys as `repo:${url}:${skillPath}`, while `RepoScanner.tsx` checked selection using `repo:${scannedUrl}:${scannedRef}:${skill.path}`. The missing `ref` segment meant `selectedSkillKeys.has(key)` was always `false` for repo skills.

## Fix Applied

**File:** `components/plugin-builder/SkillPicker.tsx`, line 275

**Before:**
```typescript
case 'repo':
  return `repo:${skill.url}:${skill.skillPath}`
```

**After:**
```typescript
case 'repo':
  return `repo:${skill.url}:${skill.ref}:${skill.skillPath}`
```

`skill.ref` is already present on `PluginSkillSelection` for repo type, so no type changes were needed. The key format now matches `RepoScanner.tsx` exactly, fixing the broken UI indicator.

- Deduplication logic in `page.tsx` (which also uses `getSkillKey`) benefits from the fix as well, as both sides now consistently include `ref`.

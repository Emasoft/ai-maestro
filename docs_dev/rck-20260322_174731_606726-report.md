## Issue Fixed

**Missing `.trim() === ''` check for `author.name` in `validateBuildConfig`**

`/Users/emanuelesabetta/ai-maestro/app/api/plugin-builder/build/route.ts`

## Root Cause

In `validateBuildConfig` (line 107-111), the `author.name` field was checked only for being a `string` (`typeof ... !== 'string'`), but unlike every other string field validated in the same function (`b.name`, `b.version`, `s.name`, `s.id`, `s.marketplace`, `s.plugin`, `s.url`, `s.ref`, `s.skillPath` — all use `.trim() === ''`), it did not reject whitespace-only strings. A payload like `{ author: { name: "   " } }` would pass validation and propagate a blank author name downstream.

## Change Applied

Before:
```typescript
  if (typeof b.author !== 'object' || b.author === null || typeof (b.author as Record<string, unknown>).name !== 'string') {
    return 'author: must be an object with a string name field when provided'
  }
```

After:
```typescript
  if (typeof b.author !== 'object' || b.author === null || typeof (b.author as Record<string, unknown>).name !== 'string' || ((b.author as Record<string, unknown>).name as string).trim() === '') {
    return 'author: must be an object with a non-empty string name field when provided'
  }
```

The comment above the block was also updated from "string name field" to "non-empty string name field" to accurately describe the now-enforced constraint.

## Verification

- All other string validations in `validateBuildConfig` and `validateSkillSelection` already use `typeof x !== 'string' || x.trim() === ''` — the fix makes `author.name` consistent with all of them.
- No other occurrences of `author.name` validation exist in the codebase that have the same omission.
- The fix is additive — it only rejects inputs that were previously silently accepted but semantically invalid.

## Bug Fixed

**BUG: Generic catch block hides specific errors / Catch block mishandles all exceptions as invalid body**
(Severity: high — reported by both gemini-2.5-flash and grok-4.1-fast)

### Root Cause

The original `catch` block used the parameterless form (`catch {`) so the caught value was inaccessible. Every thrown value — whether a `SyntaxError` from `request.json()` parsing invalid JSON, or any unexpected internal error thrown by `buildPlugin` — was unconditionally returned as HTTP 400 "Invalid request body". This:

- Masked server-side failures (database errors, filesystem errors, type errors, etc.) behind an incorrect 400 status code, misleading clients about the real cause.
- Produced no server-side log output, making debugging impossible.

### Fix Applied

**File:** `app/api/plugin-builder/build/route.ts`

The `catch` block was rewritten to:

1. Capture the error with `catch (error: unknown)`.
2. Log every error unconditionally with `console.error('Plugin build error:', error)` so all failures surface in server logs.
3. Check `error instanceof SyntaxError` — this is the precise exception thrown by the Web Fetch API's `request.json()` when the body is not valid JSON — and return 400 "Invalid request body" only in that case.
4. Return 500 "Internal server error" for every other exception, correctly signalling a server-side failure to the client.

### What Was NOT Changed

- The second reported finding (no request body size limit) was not addressed. It is a security hardening suggestion, not a correctness bug. Adding a magic-number byte limit would constitute an ad-hoc patch contrary to the project's fail-fast, no-magic-numbers coding standards.
- No other logic, imports, or types were modified.

### Verification

After the edit, the full file was re-read and confirmed correct. The catch block now reads:

```typescript
} catch (error: unknown) {
  console.error('Plugin build error:', error)
  if (error instanceof SyntaxError) {
    return NextResponse.json(
      { error: 'Invalid request body' },
```

`skill.id.split(':')[2]` accesses the third segment (index 2) of the colon-split array. If the `id` has fewer than 3 colon-separated parts (e.g., `plugin-name:skill-name` with only 2 parts), index 2 is `undefined`. The `|| skill.id` fallback catches the `undefined` but the intent is always to display the last meaningful segment of the id.

### Fix Applied

Replaced the fixed-index access with a `parts[parts.length - 1]` lookup that always retrieves the last segment, regardless of how many colon-separated parts the id contains:

**Before:**
```typescript
case 'marketplace':

```

**After:**
```typescript
case 'marketplace': {
  // Always get the last colon-separated segment, regardless of how many segments exist
  const parts = skill.id.split(':')
  return parts[parts.length - 1] || skill.id
}
```

## Fix 1 (CRITICAL): `parseMultipart` — UTF-8 corruption of text fields

**Root cause**: The original implementation decoded the entire multipart body as a latin1 string (`body.toString('latin1')`) for splitting/parsing. When the file part was extracted it was re-encoded back to a Buffer with `Buffer.from(content, 'latin1')` (byte-safe round-trip). However, the `options` text field was taken directly from the latin1-decoded string, meaning any UTF-8 multi-byte sequences (e.g. `café`) were represented as the raw bytes interpreted as individual latin1 characters (`cafÃ©`), producing invalid JSON.

**Fix applied**: Rewrote `parseMultipart` to operate byte-accurately at the Buffer level:
- Boundary search is done directly on the raw `body` Buffer using `Buffer.prototype.indexOf`
- Part headers (always 7-bit ASCII) are decoded as latin1
- File content is sliced directly from the raw Buffer — no string encoding at all, fully byte-exact
- The `options` text field is decoded from the raw Buffer slice as UTF-8 (`body.slice(...).toString('utf8')`)

This eliminates both the UTF-8 corruption of text fields and preserves binary file content without any encoding loss.

---

## Fix 2 (HIGH): `decodeURIComponent` on already-handled path params

**Root cause**: Eight route handlers called `decodeURIComponent(params.address)`, `decodeURIComponent(params.file)`, or `decodeURIComponent(params.address)` on param values extracted from the regex match of `parsedUrl.pathname`. Node.js `url.parse` does not decode the pathname, so the params may still be percent-encoded. Calling `decodeURIComponent` on a string containing a malformed percent sequence (e.g. `%GG`, or `%` not followed by two hex digits) throws `URIError: malformed URI sequence`, which crashes the handler.

**Affected handlers** (all `decodeURIComponent(params.*)` calls removed):
- `GET /api/conversations/:file/messages` — `params.file`
- `GET /api/agents/:id/amp/addresses/:address` — `params.address`
- `PATCH /api/agents/:id/amp/addresses/:address` — `params.address`
- `DELETE /api/agents/:id/amp/addresses/:address` — `params.address`
- `GET /api/agents/:id/email/addresses/:address` — `params.address`
- `PATCH /api/agents/:id/email/addresses/:address` — `params.address`
- `DELETE /api/agents/:id/email/addresses/:address` — `params.address`
- `GET /api/v1/agents/resolve/:address` — `params.address`

**Fix applied**: Removed all `decodeURIComponent()` wrapping. Params are passed directly to service functions. If percent-decoding of path segments is needed, it must be done safely (with try/catch) inside the individual service functions that require it, not in the router.

## Fix 3 (MEDIUM): `getQuery` — array-valued query params silently dropped

**Root cause**: `url.parse(url, true).query` yields `string[]` for duplicate query keys (e.g. `?mode=foo&mode=bar` produces `{ mode: ['foo', 'bar'] }`). The original `getQuery` only stored the value when `typeof v === 'string'`, silently dropping array values entirely.

**Fix applied**: Added an `Array.isArray(v)` branch that takes the first element of the array, consistent with how HTTP frameworks conventionally handle duplicate query params when a single value is expected.

```typescript

  if (v[0] !== undefined) q[k] = v[0]
} else if (typeof v === 'string') {
  q[k] = v
}
```

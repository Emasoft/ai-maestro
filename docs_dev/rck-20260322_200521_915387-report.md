# Fix Report: FID00008 BuildAction.tsx useEffect cleanup

**File**: `components/plugin-builder/BuildAction.tsx`
**Fix applied**: 2026-03-22

## Root cause

Two bugs in the original cleanup `useEffect`:

1. **Order bug (use-before-define)**: The `useEffect` referencing `clearPoll` appeared at lines 32-37, *before* `clearPoll` was declared via `useCallback` at line 39. `useCallback` assignments are not hoisted in JavaScript, so `clearPoll` was `undefined` when the effect was registered on mount.

2. **Wrong dependency array**: `[clearPoll]` in the dependency array meant the effect would re-register whenever `clearPoll` changed identity (even though it was stable via `useCallback([])`). A cleanup-only effect must use `[]` to run exactly once on unmount.

3. **Unnecessary side-effect on unmount**: `clearPoll()` resets `pollFailures.current = 0` — an operation that is meaningless when the component is being destroyed. The cleanup should only release the interval resource.

## Fix applied

- Moved `clearPoll` declaration to *before* the cleanup `useEffect` so it is in scope when the effect runs.
- Changed the cleanup `useEffect` dependency array from `[clearPoll]` to `[]`.
- Replaced the `clearPoll()` call in the cleanup function body with direct `clearInterval(pollRef.current)` + `pollRef.current = null`, which is the minimal correct resource release with no unnecessary side-effects.
- `clearPoll` (with its `pollFailures.current = 0` reset) is still used during active polling in `handleBuild` where the reset is semantically correct.

## Verification

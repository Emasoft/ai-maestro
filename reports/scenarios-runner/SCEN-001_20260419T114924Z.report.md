# Scenario Report: SCEN-001 Title Change Lifecycle

## Summary

Full end-to-end title lifecycle validation through the TitleAssignmentDialog UI. A test agent was created as AUTONOMOUS, joined a team (auto-becoming MEMBER + programmer role-plugin), then cycled through ORCHESTRATOR → ARCHITECT → MEMBER → ORCHESTRATOR with role-plugin swaps verified at every step. Per-team ORCHESTRATOR singleton enforcement (S030) was **broken** on the incoming branch — the UI dialog couldn't detect that the slot was taken because the ChangeTitle pipeline never populated `team.orchestratorId`. Fixed in Rule 4 fashion with a new Gate 13b that sets it post-write and a Gate 8 update that rejects the change pre-write. Re-verified S028-S031 against the patched backend: the second agent's dialog now correctly shows "Only one Orchestrator is allowed per team. \"scen-test-title-agent\" already holds this title." Cleanup removed the agent, the folder, and all trace from the registry.

## Bugs Found & Fixed

### BUG-001 (P1): Per-team ORCHESTRATOR singleton NOT enforced in UI or pipeline

- **Discovered at:** Step S030 (initial run)
- **Symptom:** After assigning ORCHESTRATOR to `scen-test-title-agent` in team `scen003-test-wizard-team`, opening the TitleAssignmentDialog on a second team member (`scen003-member-zeta`) showed the ORCHESTRATOR card fully enabled — normal opacity, `pointerEvents: auto`, no "Only one Orchestrator allowed" message — despite a conflicting orchestrator already existing.
- **Root cause:** Two missing pieces in `services/element-management-service.ts`:
  1. `ChangeTitle` Gate 8 only logged a note ("caller must validate team slot") instead of actually checking and rejecting on conflict.
  2. No gate anywhere in `ChangeTitle` ever WRITES `team.orchestratorId` or `team.chiefOfStaffId`. Only clearing logic existed (G11, G12, G04b, 4547). Result: after title assignment, `team.orchestratorId` remained `null`. The UI dialog reads this field to grey out the option (`components/governance/TitleAssignmentDialog.impl.tsx:291-294`), so with `null` it never showed the conflict.
- **Fix:** `services/element-management-service.ts`:
  - Gate 8 now reads `team.orchestratorId` / `team.chiefOfStaffId` for the agent's current team and refuses with `result.error = "Only one Orchestrator is allowed per team. '<name>' already holds this title in team '<team>'."` (symmetric for COS).
  - New Gate 13b (between Gate 13 and Gate 14) writes `{ orchestratorId: agentId }` or `{ chiefOfStaffId: agentId }` to the team via `updateTeam()` after a successful title change, before the agent's own `governanceTitle` is persisted in Gate 14.
- **Verified at:** S027-retest (team.orchestratorId populated in teams.json) + S030 (dialog now shows "Only one Orchestrator is allowed per team" on the second agent). `npx tsc --noEmit` passes. `yarn build` completes. PM2 restarted and both scenarios (fresh assignment + second-agent probe) succeed.
- **Files:** `services/element-management-service.ts` (~61 lines of diff)
- **Commit:** `c268b6a14d00641e528dccb562331be055dc84cc`

## Issues Noticed (Non-Blocking)

### ISSUE-1 (WARN): RBAC probes return 401 not 403

- **Noticed at:** S014 and S032
- **Severity:** WARN (informational — same as SCEN-005/006/007/008)
- **Description:** Sending `X-Agent-Id: <id>` without a Bearer token returns 401 (`auth_required`) instead of 403 (`self-modification forbidden` or `insufficient permissions`). The agent-identity layer rejects unauthenticated requests before the RBAC layer can produce the more specific denial.
- **Impact:** The security property (denial) is preserved. Scenario expectations should say "4xx denied" or match either 401 or 403.

### ISSUE-2 (WARN): "ASSIGN TITLE" vs MEMBER label divergence

- **Noticed at:** S017-S034 (throughout)
- **Severity:** WARN (cosmetic, pre-existing, repeatedly documented in SCEN-005/006/007/008)
- **Description:** When an agent's title is assigned via auto-transition on team-join, the profile panel's title button label shows "ASSIGN TITLE" even though a nearby static badge reads "MEMBER". Fix: `components/agent-profile/OverviewTab.tsx` should use `(agent.governanceTitle ?? (agent.teamId ? 'member' : 'autonomous')).toUpperCase()`.

### ISSUE-3 (INFO): Scenario expectation for S036/S037 mismatches "Also delete folder" flow

- **Noticed at:** S036
- **Severity:** INFO
- **Description:** The scenario expects the agent to appear in the Cemetery after deletion, but when the user checks "Also delete agent folder" the API is called with `hard=true`. `services/element-management-service.ts` G03 explicitly skips cemetery archive when `hard=true` ("Hard-delete — skipping cemetery archive"). The agent is therefore fully removed with no cemetery entry. The scenario file should split the final step into two paths (soft-delete-to-cemetery vs hard-delete-no-cemetery) or always use soft-delete for the cemetery probe.

### ISSUE-4 (WARN): "sudo_required" raw text briefly visible after Leave team (SCEN-006 P1-UI-2, still unfixed)

- **Noticed at:** S034
- **Severity:** WARN (cosmetic, pre-existing)
- **Description:** The profile panel briefly displayed "sudo_required" as a raw string after clicking the Leave team X. The underlying operation still succeeded once the sudo modal was re-confirmed, but the inline error leaks through because the Leave-team handler doesn't go through `sudoFetch`.

### ISSUE-5 (INFO): Help Panel positioned outside viewport when width ≤ 1280

- **Noticed at:** Initial S008 setup
- **Severity:** INFO (affects automation, not real users)
- **Description:** `components/HelpPanel.tsx` renders the panel with `class="fixed top-0 right-0 h-full w-[420px] z-50"` and a closed state that translates it by its own width. On a 1280×800 viewport (scenario-default desktop) the panel's bounding rect lands at `x=1280-1700` which is outside the visible area. `page.click` refuses to click it because "element is outside of the viewport", even though the user would see the button fine at a wider viewport. Workaround: direct DOM-level click via `evaluate`. Proper fix: `right: 0` should anchor to viewport right edge; the current left-offset seems to compute against a larger containing block.

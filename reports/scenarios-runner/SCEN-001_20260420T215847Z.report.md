## Bugs Found & Fixed (in-place, committed to feature/team-governance)

### BUG-001 (P0): AgentList.tsx SWC parser wedge — dashboard 100% unreachable
- **Discovered at:** Phase 0 SAFE-SETUP, before S004 could even render.
- **Symptom:** Every request to `http://localhost:23000/` returned HTML that
  refused to hydrate because `main-app.js` and `app-pages-internals.js`
  returned 404 (Next.js compile failed). `<LoginGate>` stayed forever on
  `Verifying session...` and no scenario could even reach its login step.
  The server log repeatedly emitted
  ```
  ./components/AgentList.tsx
  x Unexpected token `div`. Expected jsx identifier
     603 |   }
     605 |   return (
     606 |     <div className="flex flex-col h-full">
  ```
  even though the JSX at that location was visibly correct.
- **Root cause:** commit `9f46fb91` introduced
  ```ts
  const haephestosVisible = claudeProbe.loading || claudeProbe.error !== null
    ? true
    : claudeProbe.data?.available === true
  ```
  The `X || Y ? Z : W` ternary without explicit parens is legal JS (`||`
  binds tighter than `?:`), but SWC's TS-with-JSX parser wedges on this
  pattern and then, 372 lines later, fails to find a JSX identifier at the
  very first `<div>` of the component's `return(...)` block. Verified by
  (a) restoring the pre-9f46fb91 file → compile succeeds; (b) re-applying
  the feature with explicit parens → compile succeeds.
- **Fix:** `components/AgentList.tsx:228-237` — split the inline ternary
  into three explicit boolean locals so each operand is unambiguous. No
  behaviour change.
- **Committed as:** `a1107965`.
- **Verified at:** S004 (dashboard rendered login form), S006 (login
  succeeded + dashboard loaded), and every subsequent step.

### BUG-002 (P0): ORCHESTRATOR → MEMBER demotion silently strands agent at title=null
- **Discovered at:** S025 first attempt. Attempting the documented
  demotion path `ARCHITECT → MEMBER` (via the Title Assignment Dialog
  + governance password + sudo password) left the agent with
  `governanceTitle=null` in the registry while the UI's optimistic
  rendering showed `MEMBER` — a split state with no UI path to recover.
- **Symptom:** Server log showed
  ```
  PATCH /api/agents/<id> 403  (first call needs sudo)
  [ChangeTitle] ... architect → none (30 gates)
  PATCH /api/agents/<id> 200  (clear succeeded)
  PUT   /api/teams/<team-id>  400
  ```
  The 400 on PUT /api/teams aborted the flow before
  `setGovernanceTitle('member')` ever ran.
- **Root cause:**
  `components/governance/TitleAssignmentDialog.impl.tsx:471-480` handles
  ARCHITECT/INTEGRATOR/ORCHESTRATOR → MEMBER as a 2-step: clear the title
  on the agent, then (if leaving orchestrator) call
  `updateTeamOrchestratorId(null)` which PUTs
  `{orchestratorId: null}` to `/api/teams/[id]`. The route handler's
  `UpdateTeamSchema` is `.strict()` and did NOT list `orchestratorId`,
  so Zod rejected the body with HTTP 400. The service layer
  (`services/teams-service.ts:UpdateTeamParams`) and storage layer
  (`lib/team-registry.ts:updateTeam`) both already accept the field — only
  the HTTP route schema was missing it.
- **Fix:** `app/api/teams/[id]/route.ts:9-22` — added
  `orchestratorId: z.string().uuid().nullable().optional()` to
  `UpdateTeamSchema`. The field is a team-level slot, not a governance-
  gated permission, so accepting it in the general PUT body is safe.
- **Committed as:** `c6c39958`.
- **Verified at:** S025 after fix — ORCHESTRATOR → MEMBER demotion
  reached `governanceTitle=member` with the programmer plugin swapped in.

## Issues Noticed (Non-Blocking)

### ISSUE-001 (P2): /teams page delete dialog lacks "Delete Agents Too" checkbox
- **Noticed at:** S034b
- **Severity:** WARN (design gap)
- **Description:** The scenario file expects a "Delete Agents Too" checkbox
  when deleting a team via the /teams page. The current UI only offers
  Cancel / Delete Team with a governance password input and the message
  "Agents will be reverted to AUTONOMOUS and hibernated." There is no
  path from this dialog to also delete the team's agents (including the
  auto-COS). A user cleaning up after a scenario or a failed experiment
  must delete the orphan COS separately, which is annoying.
- **Suggested fix:** Add a "Also delete member agents (including auto-COS)"
  checkbox to the /teams page delete dialog that wires through
  `/api/teams/[id]?deleteAgents=true` (route already accepts this per
  `DeleteTeamSchema`).

### ISSUE-002 (P2): DeleteAgentDialog always sends hard=true — no UI path to soft-delete
- **Noticed at:** S035
- **Severity:** WARN (scenario vs code divergence)
- **Description:** `components/DeleteAgentDialog.tsx:50-76` is commented
  to explain that the UI always sends `hard=true` (SCEN-002 P0-003
  decision). The "Also delete agent folder" checkbox only controls
  `deleteFolder`. So:
  - unchecked + Delete Forever → hard-delete from registry, folder kept,
    **no cemetery archive**
  - checked + Delete Forever → hard-delete from registry + folder gone,
    **no cemetery archive**
  The scenario's S035 and S036 assume a soft-delete path that produces a
  cemetery zip. In practice the cemetery is only written by the older
  `services/agent-management-service.ts` `archiveAgentToCemetery()` call
  path, which the current DeleteAgentDialog bypasses.
- **Suggested fix:** either (a) add a "Keep a soft-deleted copy in the
  cemetery" checkbox to the dialog so users can opt into cemetery
  preservation, or (b) make the default always archive to cemetery
  (deleting the archive is cheap compared to losing agent state on a
  misclick), or (c) update the scenario file to reflect the current
  behavior — hard-delete everywhere, no cemetery from the UI.

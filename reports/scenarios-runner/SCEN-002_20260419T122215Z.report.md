## Bugs Found & Fixed

### BUG-001 (P1): hooks/useTeam.ts masks specific server errors as generic "Failed to update team"

- **Discovered at:** Step S031 (Victoria remove attempt)
- **Symptom:** When the server rejected a team update with a specific error message (e.g., R4.7 "Cannot remove the Chief-of-Staff from team members"), the UI showed "Failed to update team" instead. Users could not understand why the operation was rejected.
- **Root cause:** `hooks/useTeam.ts:82-85` threw `new Error('Failed to update team')` without reading the response body.
- **Fix:** `hooks/useTeam.ts:82-102` — try to parse response body, extract `error` or `message` field, throw that specific message. Falls back to `Failed to update team (HTTP ${status})` if body is not JSON.
- **Verified at:** Step S031 (Victoria remove attempt after fix) — client-side guard in TeamOverviewSection fires correctly showing "Cannot remove the Chief-of-Staff from team members. Unassign them first." message.
- **File:line:** `/Users/emanuelesabetta/ai-maestro/hooks/useTeam.ts:82-102`

### BUG-002 (P0): hooks/useTeam.ts sends unsupported `lastActivityAt` field causing every team update to fail

- **Discovered at:** Step S031 (Victoria remove attempt)
- **Symptom:** Server returned HTTP 400 "Validation failed" on every team PUT from the `/teams/[id]` page (team name edit, description edit, agent add/remove). Completely broke team update functionality from the team dashboard.
- **Root cause:** `hooks/useTeam.ts:80` sent `{...updates, lastActivityAt: new Date().toISOString()}` in the PUT body. The server's `UpdateTeamSchema` in `app/api/teams/[id]/route.ts:9-17` uses `.strict()` which rejects any unknown field including `lastActivityAt`. Server-side logging likely would not even have shown the specific error (Zod truncation).
- **Fix:** `hooks/useTeam.ts:76-83` — removed `lastActivityAt` from the PUT body. The `lastActivityAt` should be managed by the server, not dictated by the client.
- **Verified at:** Step S031 (retry after fix) — the client-side R4.7 guard in `TeamOverviewSection.handleRemoveAgent` fires correctly; in non-guard paths (like adding agents back in S046), the team update succeeds normally.
- **File:line:** `/Users/emanuelesabetta/ai-maestro/hooks/useTeam.ts:76-87`

### BUG-003 (P0): app/teams/page.tsx handleDelete doesn't exchange password for sudo token

- **Discovered at:** Step S057 (delete team from /teams page)
- **Symptom:** Clicking "Delete Team" on the /teams page with governance password resulted in "sudo_required" error from the server. Team deletion from the /teams page was completely broken.
- **Root cause:** `DELETE /api/teams/[id]` is classified "strict" (sudo-mode required per `security-registry.json`). The sidebar's `components/sidebar/TeamListView.tsx:78-99` correctly exchanges the inline governance password for a sudo token via `POST /api/auth/sudo-password` before the DELETE. But `app/teams/page.tsx:88-110` sent only `{password}` in the body, missing the `X-Sudo-Token` header.
- **Fix:** `app/teams/page.tsx:88-126` — exchange the password for a sudo token first via `POST /api/auth/sudo-password`, then pass BOTH the token (as `X-Sudo-Token` header) AND the password (in body) to the DELETE. Pattern mirrors the sidebar's implementation.
- **Verified at:** Step S057 (retry after fix) — team deleted successfully, agents reverted to AUTONOMOUS and hibernated.
- **File:line:** `/Users/emanuelesabetta/ai-maestro/app/teams/page.tsx:88-133`

## Issues Noticed (Non-Blocking)

### ISSUE-001 (WARN): Team dashboard lacks a consolidated "Edit Team" modal

- **Noticed at:** Phase 9 (S041)
- **Description:** The scenario file expected an "Edit Team" modal accessible via a pencil icon on the team card. The actual UI has NO edit modal — team members are added/removed individually via the team dashboard's Agents panel (Add Agent / Remove from team buttons). This differs from the scenario's expectations and from how many users may expect a bulk-edit operation to work.
- **Potential impact:** Not blocking, but UX inconsistency. Users coming from the scenario description may be confused.
- **Suggested fix:** Either rewrite the scenario to match the current UX (done by adaptation in this run), OR add an Edit Team modal to the teams page for bulk agent changes.

### ISSUE-002 (WARN): Kanban task creation fails silently when team has no GitHub project

- **Noticed at:** Phase 8 (S038)
- **Description:** Clicking "Add task" and submitting returns "Failed to create task" with no actionable hint. The underlying cause is that the team has no GitHub project linked (governance simplification 2026-03-27 made GitHub Projects the source of truth). The error should explain this clearly.
- **Potential impact:** Users will be confused when kanban doesn't work. The current error message provides no path forward.
- **Suggested fix:** Show a specific error message like "This team needs a GitHub project to store tasks. Go to Repos tab to link one." OR show a clear "Link a GitHub Project" CTA on the kanban page when `team.githubProject` is null.

### ISSUE-003 (WARN): Scenario file expects "Delete Agents Too" button in team delete dialog

- **Noticed at:** Phase CLEANUP (S057)
- **Description:** The scenario expected a "Delete Agents Too" button in the second phase of the team delete dialog. The actual UI has only "Cancel" and "Delete Team" buttons. Agents are NOT deleted by the team delete — they are reverted to AUTONOMOUS and hibernated (as indicated by the dialog text "Agents will be reverted to AUTONOMOUS and hibernated"). Agent deletion must be done separately via each agent's Advanced → Danger Zone.
- **Potential impact:** The scenario's cleanup assumption was wrong. In our run, we had to delete agents individually AFTER deleting the team.
- **Suggested fix:** Either update the scenario OR add a "Delete Agents Too" checkbox in the team delete dialog that does a batched agent delete.

### ISSUE-004 (INFO): sudo modal now has role="dialog" aria-modal="true" (improvement)

- **Noticed at:** Phase 7 (S035), Phase CLEANUP (S055, S056, S058)
- **Description:** Per MEMORY.md (2026-04-15 SCEN-020 findings), the sudo modal previously LACKED `role="dialog"` + `aria-modal="true"`. This caused the aim_sudo_modal helper to fail selector-based detection. In this run, the sudo modal correctly has these attributes, and structural detection via `aim_sudo_modal` works (with fallback if structural detection misses it).
- **Potential impact:** None — this is an improvement. The accessibility fix also makes automated test tooling more robust.

### ISSUE-005 (INFO): _aim-assistant system agent is auto-instantiated when + button clicked

- **Noticed at:** Phase 2 (S007)
- **Description:** When the + (Create new agent) button is clicked, the system auto-spawns the `_aim-assistant` helper agent (built-in help assistant owned by "system"). This creates a registry entry that persists after tests. This is by design, but it's an unavoidable side-effect of the test flow.
- **Potential impact:** Minor state drift after tests — strict STATE-WIPE (registry.json byte-for-byte match) will fail if _aim-assistant was not in the backup. In our run, backup didn't have it; post-run has it (status=offline, hibernated).
- **Suggested fix:** Either instruct scenarios to auto-delete `_aim-assistant` as part of cleanup, OR pre-spawn it in SAFE-SETUP so the backup includes it. Currently not a bug — just an awareness issue.

## Bugs Found & Fixed (Rule 4 FIX-AS-YOU-GO)

### BUG-001: TeamCreationWizard sends invalid payload fields — ALL wizard team creations fail with "Validation failed"

- **Discovered at:** S008
- **Symptom:** Clicking "Create Team" in the wizard (on `/teams` page) returned HTTP 400 with `{ error: "Validation failed" }`. The error message in the UI said only "Validation failed" with no further detail. Server logs: `POST /api/teams/create-with-project 400 in 190ms`.
- **Root cause:** `components/teams/TeamCreationWizard.tsx::handleCreate()` built a payload with fields `agentIds: []`, `autoCreateCos`, `autoCreateOrchestrator`, `githubRepos`, `newRepo`, `createGithubProject`, `githubProjectUrl` — NONE of which are accepted by `app/api/teams/create-with-project/route.ts` whose Zod schema is `.strict()` and accepts only `{ name, description, password, githubProject: {owner, repo, number}, chiefOfStaffId, orchestratorId }`. The first extra field (`agentIds`) causes the schema to reject the whole payload.
- **Fix:** `components/teams/TeamCreationWizard.tsx:278-335` — Rewrote `handleCreate()` to send only schema-accepted fields. Auto-COS is now implicit (server's `createNewTeam()` auto-creates when `chiefOfStaffId` is absent). GitHub project linking now parses `owner/repo/number` from `linkedProjectInfo.url` (previously the payload sent a non-existent field). Improved error display to include `issues[]` array from Zod response so users see which specific field is wrong. Commented what was cut (`githubRepos`, `newRepo`, `createGithubProject`) with a TODO to plumb them through the route.
- **Verified at:** S008 retry after `yarn build` + `pm2 restart` — team created in ~15s with auto-COS.

### BUG-002: `useTeam.ts` PUT body includes `lastActivityAt` which is REJECTED by the server's strict schema

- **Discovered at:** carried over from SCEN-002 run 2026-04-19; re-applied from stash during SCEN-003
- **Symptom:** Every team update from `useTeam.ts::updateTeam()` fails with HTTP 400 "Validation failed".
- **Root cause:** `hooks/useTeam.ts:74-82` added `lastActivityAt: new Date().toISOString()` to the PUT body. The server's `UpdateTeamSchema` at `app/api/teams/[id]/route.ts` uses `.strict()` and does not include `lastActivityAt` — `lastActivityAt` is a server-side concern. The extra field triggered schema rejection.
- **Fix:** `hooks/useTeam.ts:75-101` — stopped sending `lastActivityAt` in the body; also added server-error surfacing (instead of generic "Failed to update team" message, now propagates server messages like "Cannot remove the Chief-of-Staff from team members").
- **Verified at:** S040 — team delete used the same infra path; worked without "Validation failed" after fix.

### BUG-003: `/teams` page delete dialog sends `X-Sudo-Token: undefined` (sudo token never acquired)

- **Discovered at:** S040 first attempt
- **Symptom:** After typing governance password in the delete dialog and clicking "Delete Team", server returned `{ error: "sudo_required" }` 403.
- **Root cause:** `app/teams/page.tsx::handleDelete()` sent password in the request body only. `DELETE /api/teams/[id]` is a strict route classified in `security-registry.json` and requires an `X-Sudo-Token` header. The inline password in the body alone is not enough. Pattern needed: exchange the password for a sudo token via `POST /api/auth/sudo-password` first, then pass both token (header) + password (body).
- **Fix:** `app/teams/page.tsx:88-125` — added sudo-token exchange call before the DELETE, mirroring the pattern in `components/sidebar/TeamListView.tsx:78-99`.
- **Verified at:** S040 retry after rebuild — team deleted successfully.

### AUTHORING-3: S037 expects `X-Agent-Id` header alone to trigger a 403 but the auth layer rejects with 401 first

The 403 self-mod check in `lib/authorization.ts:117-122` never runs because auth fails first.
- **Adaptation:** Verified the 401 + no registry mutation. The defense is STRONGER than the scenario's 403 expectation. Documented as ISSUE-004 + proposal to update the scenario.

## Issues Noticed (Non-Blocking)

### ISSUE-001: DeleteTeam via /teams page leaves the auto-COS as orphan AUTONOMOUS agent (no "delete agents too" option)

- **Noticed at:** S040
- **Severity:** P1 (UX/cleanup)
- **Description:** When deleting a team via the `/teams` page card trash button, the dialog only offers to "revert agents to AUTONOMOUS and hibernate". There's no option to also delete the auto-created COS (`Malakai` in this run). This leaves a dangling agent with governance title stripped, team cleared, role = `chief-of-staff` (stale), and a folder on disk. Not covered by any downstream cleanup.
- **Suggested fix:** Add a "Also delete team agents" checkbox to the `/teams` page delete dialog, matching the `TeamListView.tsx` sidebar pattern. When checked, the server should cascade DELETE /api/agents/[id] for each team member (including the auto-COS).

### ISSUE-002: Delete-Agent hard-delete does NOT archive to cemetery — cemetery only holds soft-deletes

- **Noticed at:** S041
- **Severity:** INFO (by-design, but confusing)
- **Description:** My 3 agents (integrator, member, COS) were all hard-deleted (checked "Also delete agent folder"). None appear in the cemetery. The scenario expects to purge cemetery entries at S041 — but a hard-delete means no cemetery entry to begin with.
- **Suggested fix:** Either (a) update the scenario to say "IF any agents were soft-deleted, purge them here" OR (b) update the cemetery UI to explain that hard-deleted agents are not recoverable. Already by design; just a UX clarification.

### ISSUE-003: Wizard name validation debounce is flaky from synthetic events — required Playwright `page.click` via CSS/text selector

- **Noticed at:** S014 (team pick)
- **Severity:** WARN (tooling)
- **Description:** Custom React-synthetic event dispatch (via `btn.dispatchEvent(new MouseEvent('click'))`) was ignored by the React handlers in multiple places (team picker, Create Team button). Fallback to Playwright's `page.click(selector)` (which uses the real CDP Input.dispatchMouseEvent) consistently worked. This is a dev-browser scripting quirk, not an app bug.
- **Suggested fix:** Document in `aim-helpers.sh` preamble: "Always prefer `page.click(selector)` over `page.evaluate(() => btn.click())` for React buttons — React sometimes ignores synthetic events."

### ISSUE-004: Agent-identity auth layer rejects `X-Agent-Id` header without bearer token — scenario's 403 expectation outdated

- **Noticed at:** S037
- **Severity:** INFO (stronger-than-expected defense)
- **Description:** `lib/agent-auth.ts` enforces AID proof-of-possession on any request with an `X-Agent-Id` header. Without a matching `Authorization: Bearer <api-key>` (AID signature), the request is 401'd before RBAC evaluation. This is STRONGER than the scenario's expected 403 (self-mod forbidden). Result: impossible to mount a "self-mod via stolen X-Agent-Id" attack even with user session cookies.
- **Suggested fix:** Update the scenario S037 to accept EITHER 401 (auth requires bearer) OR 403 (self-mod forbidden) as success, OR add a step that obtains a valid AID bearer token first (requires `aid-init.sh` + `aid-token.sh` flow which is not yet scriptable in dev-browser sandbox).

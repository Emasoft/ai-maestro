# Scenario Report: SCEN-004 Haephestos Role-Plugin Creation Pipeline

## Summary

The Haephestos role-plugin creation pipeline was end-to-end verified: a fresh role-plugin (`scen-test-agent`) was created, the TOML preview panel rendered the profile correctly, the built plugin was published into `~/agents/role-plugins/` with `compatible-titles`/`compatible-clients` fields, it appeared in the `GET /api/agents/role-plugins` and `?title=AUTONOMOUS&client=claude-code` filtered endpoints, and cleanup removed the plugin cleanly. **However, the Haephestos agent flow itself exhibits severe usability problems** — a discovery-interview loop that cannot be skipped, PSS binary returning "Unhandled node type: string", and multi-minute context-overflow stalls that block every step. 1. Multiple authoring bugs in SCEN-004 made the scenario partially unrunnable
2. Haephestos persona is effectively unusable for rapid test runs (30+ min per plugin)
3. Several steps had to be adapted to reach the core goal

## Bugs Found

### BUG-001 (P0): Purple HELPERS Haephestos button does not navigate

**Discovered at:** S009  
**Symptom:** Clicking the purple `<button title="Haephestos — Create a role-plugin">` in the sidebar HELPERS section does nothing visible. The URL stays at `/`, main content area stays empty, and the Help/Assistant panel appears to be opened instead.  
**Root cause:** `components/AgentList.tsx:1041` uses `onClick={() => { window.location.href = '/?agent=haephestos' }}` which should trigger a full page navigation. But either (a) the button's click handler isn't firing or (b) the `app/page.tsx` `useEffect` at line 301 that processes `?agent=haephestos` fails silently. My direct navigation `page.goto('/?agent=haephestos')` also normalized the URL to `/` with no effect — the helper was never instantiated until I manually POSTed `/api/agents/creation-helper/session`.  
**Workaround:** POST `/api/agents/creation-helper/session` directly, then click the `_aim-creation-helper` card in sidebar ACTIVE.  
**Fix:** The page.tsx effect may need a check against a different dependency or the button may need `onClick` → explicit call to `setActiveAgentId(haephestosId)` + `fetch('/api/agents/creation-helper/session')`. Further investigation required.

### BUG-002 (P0): Haephestos PSS integration broken — "Unhandled node type: string"

**Discovered at:** S016  
**Symptom:** When Haephestos runs `PSS_BIN --agent ~/agents/haephestos/uploads/agent-description.md --top 12`, it returns `Unhandled node type: string`.  
**Root cause:** PSS binary error — likely a TOML-to-string serialization issue in PSS's output pipeline. Haephestos persona instructs to run PSS as Step 2, but PSS fails for simple inputs.  
**Workaround:** Have Haephestos write a hand-crafted minimal TOML directly without PSS.  
**Fix:** Reproduce with `PSS_BIN --agent <file>` against the uploaded agent-description.md, investigate PSS binary logs. Likely a PSS bug, not AI Maestro — escalate to PSS maintainers.

### BUG-003 (P0): Haephestos context overflow on multi-step operations

**Discovered at:** S016, S020, S023  
**Symptom:** During TOML generation, plugin build, and publish steps, Haephestos hit "A file being read or a tool output is likely too large for the context window" and stalled for 3+ minutes.  
**Root cause:** Haephestos loads entire repo context (CLAUDE.md=86.2k chars warning, SCENARIOS_TESTS_RULES.md=62.1k chars warning, plus ALL `.claude/rules/*.md` files) on every session. When combined with typical tool output, this rapidly exceeds the context window.  
**Workaround:** `/clear` between steps + very terse instructions. Still slow (2-3 min per command).  
**Fix:** Haephestos persona should aggressively use `/clear` between steps; or Haephestos's working directory should NOT be `ai-maestro` (which has 86.2k CLAUDE.md). Consider running Haephestos in a minimal ephemeral workdir. Also: shrink SCENARIOS_TESTS_RULES.md (it's now part of Haephestos's auto-load because it's in `.claude/rules/` symlink).

### BUG-004 (P1): No UI path to uninstall local-scope role-plugins

**Discovered at:** S030  
**Symptom:** `/api/settings/marketplaces` returns 284 marketplaces; NONE is `ai-maestro-local-roles-marketplace`. The Plugins Explorer → Marketplaces tab shows all GitHub-registered marketplaces but hides local directory-based ones. So users cannot uninstall plugins they publish through Haephestos via the UI.  
**Root cause:** `MarketplaceManager.tsx` fetches from `/api/settings/marketplaces` which is GitHub-only per `lib/marketplace-registry.ts`. The local directory marketplace at `~/agents/role-plugins/` lives in Claude CLI's settings.json but isn't surfaced by the API.  
**Workaround:** Call `DELETE /api/agents/role-plugins?name=<plugin>` directly (same endpoint the UI would call).  
**Fix:** Add local directory marketplaces to `/api/settings/marketplaces` response with `source: "local"` field, and render them with an uninstall button per plugin in `MarketplaceManager.tsx`.

### BUG-005 (P2): Scenario S030 mis-classifies DELETE /api/agents/role-plugins as strict

**Discovered at:** S030  
**Symptom:** Scenario S030 says `DELETE /api/agents/role-plugins/install` is strict (correct) but then describes deleting via a button that uses `DELETE /api/agents/role-plugins?name=...` (the "Delete Role Plugin" endpoint, not "install"). That second endpoint is NOT in `security-registry.json` strict list — it uses plain `enforceAuth` with no sudo token.  
**Root cause:** Scenario author conflated the two DELETE endpoints.  
**Workaround:** None needed — removal works without sudo.  
**Fix:** Either (a) fix scenario S030 to not expect a sudo modal for this delete path, or (b) add `DELETE_/api/agents/role-plugins` to security-registry.json as strict.

### ISSUE-001 (WARN): Help panel auto-opens on every /?agent=haephestos navigation

**Noticed at:** S009  
**Description:** After clicking the purple Haephestos card, the Help/Assistant panel slides in (top-right, 420px). The panel state persists (`translate-x-full` → closed, no translate → open) so the next time navigation fires, the panel is visible. This is incidental to BUG-001 but worth noting separately.  
**Suggested fix:** Ensure the HelpPanel doesn't auto-open on `/?agent=haephestos` route. The navigation should route directly to Haephestos without activating other UI panels.

### ISSUE-002 (WARN): Tailscale URL shown in agent list error logs

**Noticed at:** S009  
**Description:** Browser console shows `[useAgents] Failed to fetch from mac-mini-di-emanuele (http://100.99.233.43:23000)`. The error logs the Tailscale URL `http://100.99.233.43:23000` but the actual fetch uses `baseUrl=''` because `isSelf=true`. The error message is misleading — the fetch might have failed for auth/timing reasons, not Tailscale URL reasons.  
**Suggested fix:** In `hooks/useAgents.ts:102`, log the actual URL used (`baseUrl || 'same-origin'`) instead of the nominal `host.url`.

### ISSUE-003 (INFO): Creation-helper soft-deletes accumulate in registry.json

**Noticed at:** S034  
**Description:** Each POST to `/api/agents/creation-helper/session` creates a new registry entry. When cleanup runs, it marks the entry `status=deleted` but leaves it in registry.json. After multiple test runs, registry.json grows with cemetery entries (I created 3 during this test; backup had 4, end state has 7).  
**Suggested fix:** Add a periodic cleanup of creation-helper registry entries older than 24h. Or make cleanup remove the entry entirely (hard-delete) since creation-helper is explicitly `temporary: true`.

### ISSUE-004 (INFO): marketplace.json unicode encoding drift (cosmetic)

**Noticed at:** S034  
**Description:** `~/agents/role-plugins/.claude-plugin/marketplace.json` changes between `\u2014` and `—` (em dash character) across writes. This is not our concern — it's a consequence of how different writers (Node.js fs vs Python json) serialize unicode. But STATE-WIPE byte-exact match fails.  
**Suggested fix:** Use a deterministic writer (e.g. `JSON.stringify` with consistent options) across all code paths that update marketplace.json.

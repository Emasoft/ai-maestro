## Bugs Found (NOT fixed in this run)

### BUG-001 (P1): Profile panel stale selection traps title changes against wrong agent
- **Discovered at:** S018–S021
- **Symptom:** After creating `scen-test-manager` via wizard, the sidebar click on the new agent did not update the profile panel — it remained on the previously selected `scen018-mgr-v2`. Clicking the title badge then opened the title dialog for the STALE agent. My Confirm flow ended up setting `scen018-mgr-v2` → MANAGER (instead of `scen-test-manager`).
- **Root cause (inferred):** Dashboard profile panel updates only when the main-area terminal area actually swaps. The wizard's "Your Agent Is Ready!" completion dialog keeps the previous terminal view, so clicking a sidebar entry does not trigger re-selection until the wizard success dialog is explicitly dismissed via "Let's Go!".
- **Fix approach:** Either (a) close the wizard success dialog fully before allowing sidebar clicks to propagate, or (b) bind profile panel selection to the most recently clicked sidebar LI regardless of terminal state.
- **File to investigate:** `components/SessionList.tsx`, `components/AgentProfilePanel.tsx`, `app/page.tsx` wizard lifecycle.
- **Verified at:** Worked around by explicitly dismissing wizard before re-clicking agent in sidebar.

### BUG-002 (P1): Task create error swallows server message
- **Discovered at:** S041
- **Symptom:** When team has no linked GitHub Project, the kanban quick-add form shows only "Failed to create task" — the actual server message `"Cannot create task: team has no GitHub Project linked"` is hidden from the user. Users cannot self-diagnose the problem.
- **Root cause:** `hooks/useTasks.ts:76` — `if (!res.ok) throw new Error('Failed to create task')`. The server JSON body containing the informative error is never read.
- **Proposed fix:**
  ```ts
  // hooks/useTasks.ts line ~71-77
  const res = await fetch(`/api/teams/${teamId}/tasks`, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(data),
  })
  if (!res.ok) {
    let msg = 'Failed to create task'
    try { const j = await res.json(); if (j?.error) msg = j.error } catch {}
    throw new Error(msg)
  }
  ```
  And update `TaskCreateForm.tsx:49` to display `err.message` directly (already does via setError).
- **Verified at:** Not verified (not fixed).

### BUG-003 (P1): Sidebar sidebar delete-team dialog offers no "Keep Agents / Delete Agents Too" option
- **Discovered at:** S057, S071
- **Symptom:** Sidebar Teams-tab delete-team dialog shows only "Permanently delete X and revert all agents to AUTONOMOUS?" with a single Delete Team button. The scenario's expected flow (pick Keep Agents vs Delete Agents Too) is not present. Auto-COS becomes orphan AUTONOMOUS and must be deleted separately. Matches `project_cos_immutability_rules.md` MEMORY (team delete strips all titles).
- **Root cause:** The sidebar Delete Team dialog is a simpler variant than the proposed 2-option version. `/teams` page also uses the same simpler dialog.
- **Proposed fix:** Two-option dialog matching scenario (Keep Agents → revert to AUTONOMOUS; Delete Agents Too → also hard-delete the agents).
- **Files:** `components/sidebar/TeamListView.tsx`, `app/teams/page.tsx` delete handlers.
- **Verified at:** N/A.

### BUG-004 (P2): Title dialog silently closes if AUTONOMOUS already selected; Confirm stays disabled without explicit re-click
- **Discovered at:** S008 (initial attempt)
- **Symptom:** When opening the title dialog on an agent whose current title equals the pre-selected card, clicking Confirm does nothing visible (disabled). Users may mistakenly think the dialog is broken.
- **Root cause:** Dialog requires selecting a *different* card before Confirm enables (ChangeTitle is a no-op for same→same). Dialog does not message this.
- **Proposed fix:** Disable Confirm tooltip to read "Select a different title". Or let Confirm close the dialog with no-op if equal.
- **File:** `components/TitleAssignmentDialog.tsx`.
- **Verified at:** Worked around.

## Issues Noticed (Non-Blocking)

### ISSUE-001 (P2): No "MANAGER required" error when opening Create Team without a MANAGER
- **Noticed at:** S011
- **Severity:** WARN
- **Description:** The Create Team dialog opens normally even when `hasManager: false`. Submission fails silently due to other form validation (≥1 agent) without specific "MANAGER required" message.
- **Suggested fix:** Add a top-of-dialog banner reading "No MANAGER on host — teams cannot be created. Assign MANAGER first." Disable the submit button with this reason in its tooltip.

### ISSUE-002 (P3): Task creation hard-requires GitHub project
- **Noticed at:** S041
- **Severity:** INFO (by design)
- **Description:** `services/teams-service.ts:696-698` rejects all task creation for teams without a `githubProject`. The kanban UI always shows the quick-add button, inviting users to click and then fail. Matches post-2026-03-27 governance simplification.
- **Suggested fix:** Disable the "Add task" button with tooltip "Link a GitHub Project to create tasks" when `team.githubProject` is absent.

### ISSUE-003 (P2): Sudo modal can remain open and block later UI after canceled flows
- **Noticed at:** Multiple times during run (S013, S025)
- **Severity:** WARN
- **Description:** When a destructive-op chain is abandoned mid-flow, the sudo modal can linger open (one-shot token still live) and silently block the next interaction. Recovery requires Cancel or Escape.
- **Suggested fix:** Auto-dismiss sudo modal after navigation / route change / agent reselect.

### ISSUE-004 (P3): "Blocked" teams show as "R12" badge but no explicit "blocked" label
- **Noticed at:** S012
- **Severity:** INFO
- **Description:** The R12 badge is semantic to governance ruleset (R12 = MANAGER required for teams to function), but users don't know "R12 = blocked". 
- **Suggested fix:** Label the badge "BLOCKED (no MANAGER)" or add a tooltip.

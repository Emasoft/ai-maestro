## SCEN-001 — Title Change Lifecycle — **RISKY**

**File:** `/Users/emanuelesabetta/ai-maestro/tests/scenarios/SCEN-001_title-change-lifecycle.scen.md`

### Verified safe patterns
- S011-S013: Creates `scen-test-title-agent` via wizard. Name is test-prefixed and unique. Wizard creates agent in `~/agents/<name>/` by default (no "Import existing folder" path used).
- S008: Click "+" button opens the wizard with default workdir under `~/agents/` — no workdir prompt in the scenario flow.
- No `dir-fixtures`, no `Import from existing folder`, no `allowExternalFolder`.

### Risky steps

#### Line 149, S012: "Click Next through remaining wizard steps"
```
- **Action:** Click Next through team selection (skip/no team), title (leave as default), finish
```
**Risk level:** LOW but underspecified.
Relies on wizard defaults being safe. If the wizard step silently auto-selects an existing real team when "skip" is not explicit, the test agent could land in a real team. A runner must EXPLICITLY confirm "No team" is selected, not just "click Next".

#### Line 196, S017: "Cancel dialog and add agent to existing team first" — **HIGH RISK**
```
- **Action:** Cancel title dialog, click "Reassign" button next to Team field, select an existing team
```
**Risk level:** CRITICAL.
This step deliberately adds the test agent to an **arbitrary existing team**. Prerequisite line 46 says "At least 1 existing team with available slots". The runner will pick whatever team the user has. Real user teams (e.g. ecos) will then:
- Receive a rogue test agent as a new MEMBER
- May trigger governance notifications to real COS/MANAGER agents
- Dirty the team's agent roster for the cleanup window

Any failure in cleanup leaves the test agent stuck in the real team.

#### Line 284, S028: "Switch to a DIFFERENT agent in the same team" — **CATASTROPHIC**
```
- **Action:** Click on another agent that is in the same team
- **Goal:** Profile panel shows the other agent
```
**Risk level:** ABSOLUTE-NEVER-1 violation (per the audit brief).
This selects an EXISTING real user agent — whoever happens to be in the team chosen in S017. That's almost certainly a real bot like `sergei`, `barry`, `jhonny-bot`, `jack-bot`, or `genny-bot`.

#### Line 289, S029: "Open title dialog for the other agent"
Opens the Title Assignment Dialog for the real user agent.

#### Line 297, S030: "Verify ORCHESTRATOR option is DISABLED"
Visually inspects a real user agent's dialog options. Read-only at this step, but dangerous because S032 returns to the test agent using the REAL AGENT's ID.

#### Line 319, S032: "Attempt title change via API with MEMBER auth headers" — **CATASTROPHIC**
```
- **Action:** Get a MEMBER agent's ID from the same team. Send `PATCH /api/agents/<testAgentId>` with header `X-Agent-Id: <memberAgentId>` and body `{"governanceTitle": "architect"}`.
```
**Risk level:** CRITICAL.
Uses the REAL MEMBER agent's ID as the `X-Agent-Id` auth header. While the target of the PATCH is the test agent, this test impersonates a real user agent's identity. On a production governance ledger with immutable audit logs, this fabricates an audit entry under the real agent's identity — "agent X attempted title elevation of Y". The real agent is innocent, but the log now shows them attempting a forbidden operation.

### Recommendation
- S017 MUST be rewritten to create a scenario-local test team (with its own scen-test-cos) FIRST, then move the test agent into that team. Never use an existing user team.
- S028-S032 MUST use a SECOND scen-test agent created by the scenario, not an existing team member.
- S012 MUST explicitly verify "No team" is the selected option before clicking Next.

## SCEN-004 — Haephestos Role-Plugin Creation Pipeline — **RISKY**

**File:** `/Users/emanuelesabetta/ai-maestro/tests/scenarios/SCEN-004_haephestos-plugin-creation.scen.md`

### Verified safe patterns
- S011a (lines 156-161): **Explicit workdir safety check**. The scenario was designed WITH awareness of the catastrophic workdir-escape risk:
```
curl ... | jq '.agents[] | select(.name=="_aim-creation-helper") | .workingDirectory'
```
If the response is NOT under `~/agents/`, the scenario aborts with P0 finding. This is a well-designed defense against the risk the user raised.
- S003: "Ensure clean workspace" — verifies `~/agents/haephestos/` does not exist, runs `cleanup` API if it does.
- S007: Test input files written to `/tmp/scen-test-role-desc.md` and `/tmp/scen-test-project-type.md` — system scratch, safe.

### Risky steps

#### Line 136-139, S009: "Find Haephestos in the HELPERS section"
```
- **Action:** ... Click on the Haephestos agent card (`_aim-creation-helper`).
```
**Risk level:** MEDIUM.
Haephestos IS an existing system-owned agent (named `_aim-creation-helper`, matches the user's forbidden `_aim-*` pattern). BUT: Haephestos is the ephemeral forge and is DESIGNED to be selected, woken, used, and torn down. It's one of the explicit exceptions implied by the scenario's purpose. The brief's condition 6 says "any Haephestos / `_aim-*` interaction that doesn't first verify the current workdir via API" — S011a explicitly does verify workdir, so this is defended.

#### Line 149-154, S011: "Wake Haephestos"
Creates a tmux session and Claude Code process for `_aim-creation-helper`. The watchdog system (5-layer safeguards mentioned in CLAUDE.md Haephestos v2 architecture) should prevent zombie sessions. However, if the scenario crashes between S011 and S031, a Haephestos session could leak.

#### Line 330-335, S031: "Delete Haephestos agent via UI (if still present)" — **MEDIUM RISK**
```
- **Action:** ... Delete Agent. Check "Also delete agent folder". Type `_aim-creation-helper` to confirm ...
```
**Risk level:** MEDIUM-HIGH.
Deletes `_aim-creation-helper` with the "Also delete agent folder" checkbox. If the workdir check from S011a passed, this deletes `~/agents/haephestos/` which is expected. But if S011a was skipped or failed silently, this could — in theory — delete any folder the runtime has decided is the agent's workdir. The scenario DOES defend against this via S011a, so the safety depends on S011a actually running and failing loudly.

#### Line 117-122, S007: Creates files in /tmp
Creates `/tmp/scen-test-role-desc.md` and `/tmp/scen-test-project-type.md`. These are scen-test prefixed — low risk but not cleaned up by the scenario (should be explicitly purged in cleanup).

### Recommendation
- S011a is good. Keep it. Make its failure non-silent and abort-the-scenario — the scenario should explicitly say "STOP IMMEDIATELY" on mismatch (it already does on line 157).
- Add a cleanup step that explicitly rm /tmp/scen-test-*.md files to close the side-effect.
- Consider gating S031's "Also delete agent folder" checkbox on a second workdir re-verification immediately before deletion, not just on S011a which may have run dozens of steps earlier.

**Overall:** This scenario is risky BY DESIGN (it needs to touch `_aim-creation-helper` to test the forge) but the S011a defense is the right mitigation. The runner MUST NOT proceed past S011a if the workdir is outside `~/agents/`.

## SCEN-005 — Manager Gate Team Lifecycle — **RISKY**

**File:** `/Users/emanuelesabetta/ai-maestro/tests/scenarios/SCEN-005_manager-gate-team-lifecycle.scen.md`

### Verified safe patterns
- S013-S017: Creates test agent `scen-test-manager` via wizard. Test-prefixed.
- S026: Creates new test team `scen-test-governance-team`.
- S034: Creates test agent `scen-test-team-member`. Test-prefixed.
- S061: Creates test team `scen-test-blocking-team`.
- Most cleanup steps (S070-S074) target scen-test-prefixed entities by exact name.

### Risky steps

#### Line 137, S008: "If a MANAGER exists, remove MANAGER title" — **CATASTROPHIC**
```
- **Action:** Check `GET /api/governance` for `hasManager: false`. If a MANAGER exists, remove MANAGER title via title dialog (set to AUTONOMOUS with password) before proceeding.
```
**Risk level:** ABSOLUTE-NEVER-2 violation.
The user's real MANAGER agent (most likely `alexandre` based on the user's agent list) will be DEMOTED to AUTONOMOUS. This triggers the R9.8 blocking cascade:
1. All user teams get `blocked: true`
2. All team agents (bots including ecos-chief-of-staff-one, backend-infrastructure-engineer, etc.) get KILLED at tmux level (hibernated)
3. The user's entire governance state is disrupted

Even if cleanup step S070 "Re-assign MANAGER to unblock teams" restores MANAGER, the gap between S008 and S070 is the entire scenario run (~78 steps, minutes of wall time). Any crash during that window leaves the user's real infrastructure blocked.

And S070 re-assigns MANAGER to `scen-test-manager`, NOT back to the real MANAGER. The scenario assumes the real MANAGER title will be re-assignable at the end via STATE-WIPE restore, but STATE-WIPE only restores governance.json and registry.json — the actual title was removed via ChangeTitle pipeline, and ChangeTitle's side effects (plugin uninstalls, cemetery events, team-blocking-cascade log entries) are NOT trivially reverted by file overwrites.

#### Line 658-662, S075: "Delete any remaining auto-COS agents (cos-* prefix)" — **HIGH RISK**
```
- **Action:** Check agent list for `cos-` prefix agents from this test. For each, ... type the agent name and click Delete Forever.
```
**Risk level:** HIGH.
"Cos- prefix agents from this test" is ambiguous. The user has `ecos-chief-of-staff-one` which contains `cos` (though not as a prefix). More dangerously, if S075 is implemented by querying for agents where `name.startsWith('cos-')`, it could match ANY `cos-*` agent on the host — including any real user COS agent from real teams that happens to be named `cos-foo`.

The scenario relies on the runner being disciplined about "from this test" — a Haiku executor interpreting this might grab all `cos-*` matches.

#### Line 75-81, S002: Backs up 6 config files
The backup list is fine (reads only). But the restoration logic in S077 uses file-hash comparison. ChangeTitle's side effects (plugin install/uninstall, cemetery archives) are NOT captured by these file backups.

#### Line 417-421, S044: "Attempt agent lifecycle operation as wrong role"
```
- **Action:** Get MEMBER agent's ID. Attempt `POST /api/agents/<managerId>/hibernate` with header `X-Agent-Id: <memberId>`.
```
Uses `scen-test-manager` as the MANAGER (the one the scenario created) — safe. Uses `scen-test-team-member`'s ID — safe. But ASSUMES `managerId` and `memberId` are the scen-test IDs. If the runner gets confused and uses the REAL manager's ID, that hibernate attempt would target the real manager (403 response means no effect, but still an impersonation log entry).

### Recommendation
- **S008 MUST be removed or rewritten.** The scenario should either:
  - SKIP if a real MANAGER exists (with a WARN finding, not a demotion), OR
  - Use a completely isolated governance namespace (which doesn't exist yet), OR
  - Require the user to manually hibernate/remove their MANAGER before running
- **S075 MUST use explicit names** recorded in earlier steps (like S054 "Note COS agent name"), never a wildcard like "cos- prefix".

## Cross-Scenario Findings

### Pattern 1: "If real MANAGER exists, remove it" (SCEN-005 S008, SCEN-006 S007)
Both governance scenarios demote the user's real MANAGER to test the no-MANAGER blocking cascade. The cleanup attempts to restore — but only the TEST MANAGER (`scen-test-manager` / `scen006-manager`) is re-promoted, not the original. This is a **permanent state change** disguised as a test.

Mitigation: Both scenarios should require an explicit precondition "NO real MANAGER exists" and REFUSE to run if one does.

### Pattern 2: "Click an existing agent in the same team" (SCEN-001 S028)
SCEN-001 joins an existing user team AND clicks arbitrary user agents in it. This is a double violation of the 0-IMPACT rule.

Mitigation: SCEN-001 should either create a second test agent or use its own test team (like SCEN-002/003 do).

### Pattern 3: Prefix-based cleanup (SCEN-005 S075, SCEN-006 S031)
"Delete agents with cos- prefix" is fundamentally unsafe. It relies on the runner understanding "from this test" — which is not enforceable by automation.

Mitigation: All cleanup steps MUST delete agents BY EXPLICIT NAME recorded earlier in the scenario.

### Pattern 4: Haephestos safety defense (SCEN-004 S011a) — GOOD
SCEN-004 includes a workdir verification step with abort-on-mismatch before any folder-delete operation. This is the correct defensive pattern and should be propagated to any other scenario that touches `_aim-*` agents.

### Pattern 5: Ambiguous "Click Next through remaining steps" (SCEN-001 S012)
Several scenarios abstract wizard step navigation as "click Next". This presumes safe defaults. Any wizard change that introduces a new destructive default (e.g. "Import existing folder" becoming the new Step 3) would silently break every scenario using this pattern.

Mitigation: Every wizard step MUST be enumerated with explicit field values, not "click Next".

### Pattern 6: No Haephestos cleanup /tmp files (SCEN-004 S007)
SCEN-004 creates /tmp/scen-test-*.md files but never deletes them. Low impact (they're in /tmp which is system-managed), but violates CLEAN-AFTER-YOURSELF (Rule 1).

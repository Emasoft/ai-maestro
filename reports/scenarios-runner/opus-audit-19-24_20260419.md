## SCEN-022: MANAGER performs autonomous config ops

**VERDICT: RISKY**

### Risky steps:

- **S002 (lines 81-89):** "Confirm a MANAGER agent is visible in the sidebar (red TITLE badge) ... If no MANAGER, create one (not counted as scenario step but documented in prerequisites)."
  - **Risk:** The current registry has NO agents with governance titles assigned (verified: `jq '.[] | select(.governanceTitle != null)'` returned empty). The prerequisite "MANAGER agent exists" at line 52 is currently FALSE. The scenario's fallback "create one as setup step" is undocumented, ambiguous, and happens BEFORE the scenario numbered steps — meaning the runner must ad-hoc create a MANAGER agent. If the runner interprets this as "promote an existing agent to MANAGER", it could promote one of the user's 17 real agents (alexandre, luckas-bot, etc.) to MANAGER, modifying real user state. Even if it creates a new agent, the creation is outside the scenario's numbered step sequence and therefore outside cleanup.

- **S003 (lines 92-99):** "Click MANAGER card in sidebar; if hibernated, click Wake."
  - **Risk:** Given there is no MANAGER in the registry, this step has no valid target. If the runner selects an agent with `governanceTitle: "manager"` by visual scan (e.g. a red badge), there is none. If the runner selects any visible agent to try, they may end up clicking user's real agents (e.g. `alexandre` or `backend-infrastructure-engineer`) — direct violation of audit condition #2.

- **S004 (lines 105-112):** "In MANAGER's prompt builder, send: 'Create a new AUTONOMOUS agent named scen022-autobot...'"
  - **Risk:** This step sends a NL PROMPT to the MANAGER agent's terminal. If the MANAGER is a user's real agent (created via the ambiguous setup step), the scenario is MESSAGING A USER'S REAL AGENT — direct violation of audit condition #2.
  - Even if the runner creates a brand-new scen-prefixed MANAGER agent via the ambiguous setup step, the scen-prefixed agent will not be deleted by cleanup (the cleanup only deletes `scen022-autobot`, not the setup-MANAGER). This leaves an orphan test MANAGER behind that keeps leaking future governance state.

- **S006 (lines 146-154), S008 (lines 173-181), S009 (lines 184-191), S011 (lines 212-234):** These all send NL PROMPTS to the same MANAGER, compounding the above risk.

- **S010 (lines 197-205):** "Open the human user card in the sidebar, watch the inbox" — this is not an agent interaction, but the concept of a "human user card" also depends on context that may not exist in the current system.

### Additional notes:

- S004a (line 128-132) is a GOOD safety check — explicitly verifies `workingDirectory` is `/Users/<user>/agents/scen022-autobot` and aborts if not. This is solid defence in depth.
- S013 cleanup (lines 252-264) only deletes `scen022-autobot`, leaving the ambiguous MANAGER agent in the system.

### Specific remediation recommendations

**SCEN-019 S014:** Change "For each currently-running agent in the dashboard" to a specific scoped check. Safer options:
- (a) Verify the user-scope install by reading `~/.claude/settings.json` directly (Rule 6 read-only file check), which is an explicit state-verification read and does not require touching user's real agents.
- (b) Use the `/api/agents` API response + `/api/agents/<id>/local-config` directly for each agent without opening their Profile tab (pure state-verification reads per Rule 6, no UI clicks on real agents).
- (c) Restrict the check to a single scen-prefixed test agent created earlier in the scenario — but SCEN-019 creates no test agents, so the scenario would need a new step to create `scen019-verify-01` just for this check and delete it at cleanup.

**SCEN-022 S002/S003/S004:** Add explicit unambiguous setup:
- (a) Rename the prerequisite agent from "a MANAGER agent" to an explicit `scen022-mgr-setup` that the scenario itself creates in a new Phase 0 step and deletes in the cleanup phase, so the MANAGER used is always a scen-prefixed test agent.
- (b) Add S004a-style safety checks: after clicking the MANAGER card, verify via `/api/agents` that the selected agent's name starts with `scen022-` before sending any prompts. Abort immediately if not.
- (c) Refuse to run the scenario if no scen-prefixed MANAGER exists (don't fall back to "create one as setup step" — that's a scenario-step bypass of Rule 7 SAFE-SETUP).
